## 1. 전체 아키텍처 — React + Spring Boot 분리 구조

### 결정
프론트엔드(React, 포트 3000)와 백엔드(Spring Boot, 포트 8080)를 완전히 분리하는 SPA + REST API 구조를 채택했습니다.

### 이유
- **역할 명확히 분리**: UI 렌더링은 React, 비즈니스 로직과 데이터는 Spring Boot가 전담합니다.
- **배포 유연성**: 프론트(React → Nginx 정적 파일)와 백엔드(Spring Boot → Cloud Run)를 각각 독립적으로 배포·스케일링할 수 있습니다.
- **나중에 앱으로 확장 가능**: REST API 구조이므로, 이후 모바일 앱(Flutter 등)을 만들어도 같은 API를 그대로 사용할 수 있습니다.

## 2. Spring Security 미사용 — HandlerInterceptor 기반 JWT 인증

### 결정
Spring Security를 완전히 제거하고, `JwtInterceptor` (HandlerInterceptor 구현)로 JWT 인증을 직접 구현했습니다.

### 이유
- **학습 목적**: Spring Security는 내부 동작이 복잡하고 설정이 많아 팀원이 코드를 이해하기 어렵습니다. 인터셉터 방식은 요청이 들어오는 흐름을 코드에서 직접 추적할 수 있습니다.
- **불필요한 복잡성 제거**: 이 프로젝트의 인증 요구사항은 "로그인 여부 확인 + 관리자 여부 확인" 두 가지뿐입니다. 이를 위해 Spring Security 전체를 도입할 이유가 없었습니다.
- **선택적 인증 처리가 쉬움**: 공개 경로에서도 토큰이 있으면 사용자 정보를 추출하는 "선택적 인증" 로직을 인터셉터에서 직접 처리했습니다. Spring Security로 구현하면 복잡한 설정이 필요합니다.
- **BCrypt 직접 사용**: `org.mindrot:jbcrypt:0.4`를 독립 라이브러리로 사용해 Spring Security 의존성 없이도 비밀번호 해싱이 가능합니다.

### 동작 방식
```
요청 → JwtInterceptor.preHandle()
  ├── OPTIONS 요청이면 → 통과 (CORS preflight)
  ├── PUBLIC_PATHS에 해당하면 → 통과 (단, 토큰 있으면 사용자 정보 추출)
  ├── 토큰 없거나 유효하지 않으면 → 401 반환
  ├── /api/admin/** 인데 ADMIN 아니면 → 403 반환
  └── 정상이면 → request에 "쉼표번호", "role" 저장 후 컨트롤러로 전달
```

## 3. 쉼표번호 — String PK 선택

### 결정
사용자 식별자(PK)로 `쉼표0001` 형태의 VARCHAR(12) String을 사용합니다.

### 이유
- **도메인 의미 부여**: 단순한 숫자 ID 대신 서비스 브랜드(쉼표)가 담긴 식별자를 사용해, 사용자에게 닉네임으로도 노출할 수 있습니다.
- **초기 가입자 경험**: 첫 가입자는 `쉼표0001`이 됩니다. 숫자 1번 유저라는 특별함을 닉네임 형태로 표현할 수 있습니다.
- **현실적인 결정**: MVP 단계에서 PK를 BIGINT로 분리하는 정석 설계는 34개 FK 테이블 수정이 필요합니다. 이를 무리하게 진행하면 개발이 지연되므로, MVP 완성 후 리팩토링으로 미뤘습니다.

### 주의사항
- `쉼표번호`는 절대 BIGINT로 변환하지 않습니다 (DB 설계 구조상 불가).
- 사용자에게 보여주는 닉네임은 `nickname` 컬럼을 별도로 사용합니다 (변경 가능).

## 4. MyBatis 선택 — JPA/Hibernate 대신

### 결정
ORM으로 JPA/Hibernate 대신 MyBatis(XML 기반 SQL 매퍼)를 사용합니다.

### 이유
- **SQL 직접 제어**: 쉼표 서비스는 진단 점수 계산, 날씨 기반 추천 정렬, 월간 통계 집계 등 복잡한 SQL이 많습니다. MyBatis는 SQL을 그대로 작성하므로 쿼리 최적화가 쉽습니다.
- **학습 곡선**: JPA는 영속성 컨텍스트, N+1 문제, JPQL 등 추가 개념이 많습니다. MyBatis는 SQL을 알면 바로 사용할 수 있어 팀 전체가 코드를 이해하기 쉽습니다.
- **동적 쿼리 명시적 관리**: `<if>`, `<foreach>` 등 MyBatis의 동적 쿼리 태그로 조건부 SQL을 명시적으로 관리합니다. JPA Specification보다 직관적입니다.

## 5. JWT 토큰 — Access + Refresh 이중 구조

### 결정
- **Access Token**: 짧은 만료 시간 (설정값, 기본 1시간), Authorization 헤더로 전달
- **Refresh Token**: 긴 만료 시간 (설정값, 기본 7일), Redis에 저장

### 이유
- **보안과 편의의 균형**: Access Token 만료 시간을 짧게 하면 보안이 좋지만 자주 로그아웃됩니다. Refresh Token으로 Access Token을 자동 갱신해 사용자 경험을 보장합니다.
- **Redis에 Refresh Token 저장**: DB가 아닌 Redis에 저장해 만료·삭제가 빠릅니다. 로그아웃 시 Redis에서 삭제하면 토큰을 즉시 무효화할 수 있습니다.
- **프론트엔드 자동 갱신**: `fetchWithAuth.js`에서 401 응답을 감지하면 자동으로 Refresh Token으로 갱신 후 원래 요청을 재시도합니다. 여러 요청이 동시에 401이 나도 큐(refreshQueue)로 관리해 중복 갱신을 방지합니다.

## 6. 소셜 로그인 — Pending Token 방식

### 결정
카카오/구글/네이버 OAuth2 로그인 시, 신규 사용자는 바로 계정을 만들지 않고 "가입 확인 페이지"로 이동시킵니다. 이때 사용자 정보를 임시 JWT(pending token)에 담아 10분간 유지합니다.

### 이유
- **username 직접 설정 허용**: 소셜 계정 정보만으로는 사용자가 원하는 로그인 아이디(username)를 알 수 없습니다. 확인 페이지에서 사용자가 직접 username을 입력합니다.
- **세션 대신 JWT 사용**: 서버에 임시 세션을 만들지 않고 JWT에 소셜 정보를 담아 stateless를 유지합니다.
- **기존 계정 연동 구분**: 기존 쉼표 계정에 소셜을 연동하는 경우(link)와 신규 가입(signup)을 JWT state 파라미터로 구분합니다.

## 7. 추천 알고리즘 — 날씨·시간대 가중치 보정

### 결정
진단 점수(0~100점)만으로 추천하지 않고, 현재 날씨 상태와 시간대에 따라 추천 점수를 보정합니다.

### 이유
- **맥락 없는 추천은 신뢰도 저하**: 비 오는 날 "공원 산책" 을 1순위로 추천하면 사용자는 앱을 신뢰하지 않습니다.
- **실제로 실행 가능한 추천**: 현재 날씨(맑음/비/흐림)와 시간대(저녁)를 반영해 지금 바로 할 수 있는 휴식을 추천합니다.

### 가중치 예시
| 상황 | 올라가는 유형 | 내려가는 유형 |
|------|-------------|-------------|
| 비 오는 날 | 감각적(+15), 창조적(+15), 정신적(+10) | 자연적(-20), 신체적(-20) |
| 맑은 날 | 자연적(+15), 신체적(+10) | — |
| 저녁 | 사회적(+20), 감각적(+10) | — |

## 8. 원천데이터 vs 산출물 분리

### 결정
심박수 측정값(`heart_rate_measurements`)과 설문 응답(`survey_responses`)은 원천데이터로 별도 보관하고, 진단 결과(`diagnosis_results`)는 이를 계산한 산출물로 분리합니다.

### 이유
- **재계산 가능**: 점수 계산 로직이 바뀌어도 원천데이터가 있으면 과거 진단을 다시 계산할 수 있습니다.
- **감사 가능성(Auditability)**: "이 진단 결과가 어떤 측정값에서 나왔는가"를 추적할 수 있습니다.
- **분석 확장성**: 나중에 ML 모델을 적용할 때 원천데이터를 학습 데이터로 바로 사용할 수 있습니다.

## 9. analytics_events — @Async 비동기 수집

### 결정
사용자 행동 이벤트(페이지뷰, 기록 생성, 진단 완료 등)를 `@Async`로 비동기 처리합니다.

### 이유
- **메인 흐름 지연 방지**: 이벤트 DB 저장이 실패하거나 느려도 사용자의 실제 요청(기록 저장, 진단 완료)은 영향받지 않습니다.
- **관심사 분리**: 비즈니스 로직(기록 저장)과 통계 수집을 분리합니다. `AnalyticsService.track()`을 호출하면 즉시 반환되고, 실제 DB 저장은 별도 스레드에서 처리됩니다.
- **장애 격리**: 이벤트 수집이 실패해도 try-catch로 조용히 무시합니다. 수집 기능 장애가 서비스 장애로 이어지지 않습니다.

## 10. 금칙어 — 소프트 필터링 (shadow hide)

### 결정
금칙어가 포함된 게시글/댓글을 거부하지 않고, `status = 'hidden'`으로 저장합니다. 작성자 본인은 볼 수 있지만 다른 사용자에게는 보이지 않습니다.

### 이유
- **사용자 경험**: 거부 메시지를 받으면 사용자가 어떤 단어가 금칙어인지 역으로 파악할 수 있습니다. 소프트 히든 방식은 이를 방지합니다.
- **관리자 검토 가능**: 숨겨진 게시글도 DB에 남아있으므로, 관리자가 나중에 검토해 복원하거나 완전 삭제할 수 있습니다.
- **무료 운영**: 외부 AI API나 유료 필터링 서비스 없이 `blocked_keywords` 테이블만으로 운영합니다.

## 11. 신고 기능 — 중복 신고 차단

### 결정
같은 사용자가 같은 대상을 중복 신고할 수 없도록 DB에서 체크합니다.

### 이유
- **도배 방지**: 악의적인 사용자가 특정 게시글에 반복 신고를 보내 관리자 업무를 마비시키는 것을 방지합니다.
- **공정한 신고 처리**: 한 사람이 여러 번 신고해도 1건으로 처리됩니다.

## 12. VWorld POI API — 장소 Seed 데이터

### 결정
장소 데이터를 직접 입력하지 않고, 국토부 VWorld API에서 POI(Point of Interest) 데이터를 받아 자동으로 DB에 채웁니다.

### 이유
- **실제 데이터 확보**: 공공데이터를 사용하면 장소명, 주소, 위경도가 정확합니다.
- **무료**: VWorld API는 공개 무료 서비스입니다.
- **운영 편의**: POST /api/admin/places/seed를 호출하면 자동으로 데이터가 채워지므로, 수동 입력 없이 초기 데이터를 구성할 수 있습니다.

## 13. recommendations 테이블 — 반드시 저장

### 결정
사용자에게 추천을 표시할 때 반드시 `recommendations` 테이블에 기록합니다 (추천 기준, 클릭 여부, 저장 여부 포함).

### 이유
- **통계의 근거 데이터**: "어떤 장소가 많이 클릭됐는가", "어떤 휴식 유형이 저장률이 높은가"를 분석하려면 추천 로그가 필수입니다.
- **추천 품질 개선**: 클릭률/저장률 데이터를 축적하면 이후 추천 알고리즘을 데이터 기반으로 개선할 수 있습니다.
- **멘토링 지적 사항**: "추천 로그가 없으면 통계 화면이 가짜가 된다"는 멘토링 피드백을 반영했습니다.

## 14. 파일 업로드 — 로컬 저장소

### 결정
사진 업로드 파일은 서버 로컬 디렉토리(`uploads/`)에 저장하고, Spring Boot의 정적 리소스 핸들러로 서빙합니다.

### 이유
- **배포 전 개발 단계**: Cloud Storage(S3, GCS) 연동은 배포 시 추가합니다. 개발 단계에서는 로컬 저장이 설정이 없어 빠릅니다.
- **경로 일관성**: `/uploads/community/`, `/uploads/challenge/` 등 용도별 디렉토리를 분리해 파일 관리가 명확합니다.

## 15. 커뮤니티/챌린지 — 2차 MVP로 분리한 이유

### 결정
초기에는 커뮤니티, 챌린지를 2차 MVP로 분류해 핵심 흐름(진단→추천→기록→개선) 완성에 집중했습니다.

### 이유
- **멘토링 피드백**: "화면은 많은데 핵심 흐름의 데이터가 얕다"는 피드백을 받았습니다. 먼저 진단→추천→기록→개선 흐름의 데이터 품질을 완성하는 것이 우선이었습니다.
- **실제 가치**: 사용자가 쉼표에 돌아오는 이유는 휴식 기록과 통계입니다. 커뮤니티는 이 핵심 가치를 검증한 후 추가하는 것이 맞습니다.
- **결과**: 핵심 흐름 완성 후 커뮤니티, 챌린지, 신고, 금칙어, analytics까지 모두 구현했습니다.

## 16. 모바일 반응형 미구현

### 결정
데스크탑(1280px 기준) 전용으로 개발하고, 모바일 반응형은 구현하지 않았습니다.

### 이유
- **개발 기간 제한**: 한정된 시간 내에 핵심 기능을 완성하는 것이 우선이었습니다.
- **발표용 시연 환경**: 프로젝트 발표는 데스크탑 화면으로 진행하므로, 모바일 반응형이 당장 필요하지 않았습니다.
- **추후 적용 가능**: Tailwind CSS를 사용하므로 `sm:`, `md:`, `lg:` 브레이크포인트만 추가하면 반응형 적용이 가능합니다.
