## 1. 전체 아키텍처 선택 이유

### 1-1. React + Spring Boot 분리 구조 (SPA + REST API)

**선택한 것:** 프론트엔드(React)와 백엔드(Spring Boot)를 완전히 분리한 구조

**왜:**
- 팀 분업이 가능하다. 프론트 담당자는 백엔드 서버 없이도 목업 데이터로 UI를 개발할 수 있다.
- 나중에 앱(React Native)이나 다른 클라이언트가 생겨도 같은 API를 쓸 수 있다.
- 배포를 독립적으로 할 수 있다. 프론트는 정적 파일(Nginx), 백엔드는 Cloud Run으로 따로 배포한다.
- 현재 `mobile/` 폴더가 이미 있는 것처럼, 향후 모바일 확장을 위한 구조다.

**대안과 비교:**
- `Next.js (SSR)` — SEO가 중요하지 않고 관리자 페이지가 많아서 불필요
- `Spring MVC + Thymeleaf` — 프론트 개발자와 협업이 어렵고 현대적이지 않음
- `모놀리식` — 초기엔 빠르지만 배포/확장 시 전체를 다시 빌드해야 함

---

### 1-2. 포트 구분 (Frontend 3000, Backend 8080, YOLO 8090)

**왜:**
- 개발 중에 세 서버를 동시에 띄워서 독립적으로 개발/테스트할 수 있다.
- React 개발서버(3000)에서 Spring Boot(8080)로 API 요청 시 CORS 문제가 생기는데, 이를 CORS 설정으로 해결했다.
- YOLO 서비스(8090)는 Python FastAPI로 별도 프로세스다. Spring Boot가 꺼져도 YOLO가 살아있고, YOLO가 꺼져도 Spring Boot는 정상 동작한다 (graceful degradation).

## 2. 백엔드 설계 이유

### 2-1. 도메인 패키지 구조 (domain/{도메인}/controller+service+mapper+model)

```
backend/src/main/java/com/comma/
├── domain/
│   ├── auth/
│   │   ├── controller/   ← AuthController.java
│   │   ├── service/      ← AuthService.java
│   │   ├── mapper/       ← AuthMapper.java
│   │   └── model/        ← LoginRequest.java 등
│   ├── user/
│   ├── survey/
│   └── ... (15개 도메인)
└── global/
    ├── config/           ← JWT, CORS, Interceptor
    ├── exception/        ← GlobalExceptionHandler
    ├── scheduler/        ← PlaceScoreScheduler
    └── util/             ← ApiResponse, MailService 등
```

**왜:**
- **기능(도메인) 중심으로 묶었다.** 전통적인 `controller/`, `service/`, `mapper/` 레이어 중심 구조는 파일이 많아지면 어떤 기능인지 찾기 어렵다.
- 도메인 중심 구조는 `auth` 기능을 수정할 때 `auth/` 폴더 안에서만 작업이 끝난다. 다른 도메인에 영향을 주지 않는다.
- 신규 도메인 추가 시 폴더를 하나 만들고 4개 파일(Controller, Service, Mapper, XML)만 추가하면 된다.
- `global/`은 도메인에 속하지 않는 공통 코드(JWT, CORS, 응답 포맷 등)를 모아둔 곳이다.

**4세트 파일을 항상 같이 만든 이유:**
```
Controller → Service → Mapper(인터페이스) → Mapper.xml(SQL)
```
- **Controller**: HTTP 요청/응답만 담당. 비즈니스 로직 없음.
- **Service**: 비즈니스 로직. 트랜잭션 처리. Mapper를 조합해서 사용.
- **Mapper (인터페이스)**: MyBatis가 구현체를 자동 생성. SQL과 Java 사이의 계약.
- **Mapper.xml**: 실제 SQL. 복잡한 쿼리를 Java 코드와 분리해서 관리.

각 파일이 하나의 역할만 담당하게 해서 단일 책임 원칙을 지켰다.

---

### 2-2. MyBatis를 선택한 이유 (JPA 대신)

**왜:**
- **SQL을 직접 제어해야 했다.** 이 프로젝트는 복잡한 집계 쿼리(월간 통계, 진단 점수 계산, 장소 추천 로직)가 많다. JPA는 단순 CRUD엔 좋지만 복잡한 JOIN, GROUP BY, CASE WHEN 등은 `@Query`나 `QueryDSL`을 써야 해서 오히려 복잡해진다.
- **팀원들이 SQL에 익숙하다.** MyBatis는 SQL을 그대로 쓰기 때문에 쿼리를 직접 확인/수정하기 쉽다.
- **XML에 SQL을 분리**해두면 쿼리 최적화가 필요할 때 Java 코드 건드리지 않고 XML만 수정하면 된다.
- `underscore-to-camelCase` 자동 변환 설정(`map-underscore-to-camel-case: true`)으로 `like_count` → `likeCount` 자동 매핑이 된다.

---

### 2-3. Spring Security를 쓰지 않은 이유

**왜:**
- Spring Security는 강력하지만 **설정이 복잡**하다. SecurityFilterChain, UserDetailsService, GrantedAuthority 등 개념이 많아서 팀 전체가 이해하는 데 시간이 걸린다.
- 이 프로젝트의 인증 요구사항은 단순하다: **JWT가 있으면 통과, 없으면 401, ADMIN이면 관리자 API 허용.** 이 정도는 `HandlerInterceptor` 하나로 충분하다.
- `JwtInterceptor.java` 하나에 인증 로직이 모두 담겨 있어서 **어디서 어떻게 인증이 일어나는지 한눈에 보인다.**
- BCrypt는 `org.mindrot:jbcrypt:0.4`를 직접 import해서 쓴다. Spring Security 없이도 암호화가 된다.

---

### 2-4. `request.setAttribute("쉼표번호", 쉼표번호)` 패턴

**코드:**
```java
// JwtInterceptor.java
String 쉼표번호 = jwtUtil.extract쉼표번호(token);
request.setAttribute("쉼표번호", 쉼표번호);

// AuthController.java
String 쉼표번호 = (String) request.getAttribute("쉼표번호");
```

**왜:**
- JWT에서 꺼낸 사용자 ID를 Controller까지 전달해야 한다.
- `HttpServletRequest`는 요청 하나의 생명주기 동안 유지되는 객체다. 인터셉터에서 값을 담으면 Controller에서 꺼낼 수 있다.
- Spring Security의 `SecurityContextHolder.getContext().getAuthentication()`보다 **훨씬 단순하고 직관적**이다.
- 이 값은 인터셉터가 이미 검증한 값이므로 Controller에서는 신뢰하고 바로 쓸 수 있다.

---

### 2-5. OGNL 한글 파라미터 금지 — `@Param("commaNo")` 사용

**문제:**
```java
// 위험 — 한글 파라미터명은 OGNL 표현식에서 인식 안 될 수 있음
User findBy쉼표번호(String 쉼표번호);

// 안전 — 영문 별칭으로 명시
User findBy쉼표번호(@Param("commaNo") String 쉼표번호);
```

**왜:**
- MyBatis의 OGNL 표현식(`#{쉼표번호}`)은 내부적으로 리플렉션을 사용한다. 한글 파라미터명이 JVM 버전이나 빌드 환경에 따라 제대로 인식되지 않는 케이스가 있다.
- `@Param("commaNo")`로 명시하면 XML에서 `#{commaNo}`로 안전하게 쓸 수 있다.
- 실제로 이 문제를 겪고 나서 규칙으로 확립했다.

## 3. 프론트엔드 설계 이유

### 3-1. axios 대신 fetch를 쓴 이유

**왜:**
- `fetch`는 브라우저 내장 API다. 별도 라이브러리를 설치할 필요가 없다.
- `fetchWithAuth.js`라는 wrapper를 만들어서 **JWT 자동 첨부, 401 시 토큰 자동 갱신** 로직을 한 곳에 모았다.
- axios를 쓰면 interceptor 설정이 필요한데, fetchWithAuth wrapper가 그 역할을 대신한다.
- 팀원들이 `fetch()`는 이미 알고 있어서 학습 비용이 없다.

**fetchWithAuth.js의 핵심 로직:**
```javascript
// 1. API 호출
// 2. 401 응답이면 → refreshToken으로 accessToken 재발급
// 3. 재발급 성공 → 원래 요청 재시도
// 4. 재발급 실패 → localStorage 비우고 /login으로 이동
// 5. 동시에 여러 요청이 401을 받아도 refresh는 딱 1번만 실행 (큐 처리)
```

이 로직이 없으면 accessToken이 만료될 때마다 사용자를 강제 로그아웃시켜야 한다.

---

### 3-2. Tailwind CSS를 쓴 이유

**왜:**
- 별도 CSS 파일을 만들지 않아도 된다. JSX 안에서 바로 스타일을 적용한다.
- 디자인 시스템(색상, 간격, 폰트)을 `tailwind.config.js`에 한 번만 정의하면 전체 프로젝트에서 일관되게 쓸 수 있다.
- `primary`, `soft-mint`, `text-main`, `text-muted` 같은 커스텀 색상을 정의해두어서 어디서든 `bg-primary`, `text-text-main`으로 쓸 수 있다.
- 클래스명이 스타일을 직접 설명하기 때문에 CSS 파일을 왔다갔다 볼 필요 없다.

**확립된 패턴:**
```
배경: bg-white 또는 bg-[#F7F7F8]
인풋: border-b-2 언더라인 스타일
버튼: py-4 rounded-2xl font-bold shadow-lg shadow-emerald-100
카드: bg-white rounded-2xl shadow-sm border border-gray-100 p-6
타이틀: text-[28px] font-extrabold tracking-tight
```
이 패턴들이 확립된 이유는 Toss/카카오 수준의 완성도 있는 UI를 빠르게 만들기 위해서다.

### 3-3. 공통 컴포넌트 (Button, Input, Card, Toast)를 만든 이유

```
components/common/
├── Button.jsx   ← variant: primary / outline / ghost
├── Input.jsx    ← icon 지원, 에러 메시지 표시
├── Card.jsx     ← 기본 카드 레이아웃
└── Toast.jsx    ← success / error / info, 3초 자동 dismiss
```

**왜:**
- 버튼 스타일을 40개 페이지에서 각자 쓰면, 나중에 버튼 색상 하나 바꿀 때 40개 파일을 다 수정해야 한다.
- 공통 컴포넌트를 만들어두면 `<Button variant="primary">로그인</Button>` 한 줄로 쓰고, 스타일 변경은 `Button.jsx` 하나만 수정하면 된다.
- 에러 처리, 로딩 상태, 접근성 속성 등을 한 곳에서 관리할 수 있다.

### 3-4. PrivateRoute를 만든 이유

```jsx
// App.jsx
<Route path="/my" element={<PrivateRoute><MyPage /></PrivateRoute>} />
<Route path="/admin" element={<AdminRoute><AdminDashboard /></AdminRoute>} />
```

**왜:**
- 로그인하지 않은 사용자가 `/my`에 접근하면 자동으로 `/login`으로 보내야 한다.
- 각 페이지 컴포넌트 안에서 토큰 체크를 하면 코드가 중복된다. `PrivateRoute` 하나가 이 역할을 대신한다.
- 관리자가 아닌 사용자가 `/admin`에 접근하면 차단한다.
- 인증 로직이 라우팅 계층에서 한 번에 처리된다.

---

### 3-5. 페이지 구조 — user/ vs admin/ 분리

```
pages/
├── user/
│   ├── auth/
│   ├── diagnosis/
│   ├── recommend/
│   └── ...
└── admin/
    ├── users/
    ├── contents/
    └── ...
```

**왜:**
- 사용자 기능과 관리자 기능은 완전히 다른 UI/UX를 가진다.
- 분리하면 사용자 페이지를 수정할 때 관리자 페이지에 영향이 없다.
- 나중에 관리자 페이지만 별도 서브도메인(`admin.comma.com`)으로 분리할 때 용이하다.

---

### 3-6. 각 페이지에서 loading/error 상태를 항상 포함한 이유

```jsx
const [loading, setLoading] = useState(false);
const [error, setError] = useState(null);

try {
  setLoading(true);
  const res = await fetchWithAuth('/api/rest-logs');
  setLogs(res.data);
} catch (e) {
  setError('데이터를 불러오지 못했어요.');
} finally {
  setLoading(false);
}
```

**왜:**
- API 호출에는 **항상 실패 가능성**이 있다. 서버가 꺼져 있거나, 네트워크가 끊기거나, 인증이 만료됐거나.
- 로딩 중에 빈 화면을 보여주면 사용자는 앱이 망가진 줄 안다. `loading` 상태로 스켈레톤이나 스피너를 보여준다.
- 에러가 나도 앱이 완전히 멈추지 않게 `try/catch`로 잡아서 사용자에게 안내 메시지를 보여준다.
- `finally`에서 `setLoading(false)`를 해야 성공/실패 모두에서 로딩이 끝난다.

## 4. 인증 시스템 설계 이유

### 4-1. JWT Access Token + Refresh Token 이중 구조

**왜:**
- **Access Token만 쓰면:** 토큰을 길게 설정하면 탈취당했을 때 위험하고, 짧게 설정하면 자주 로그아웃된다.
- **이중 구조 해결:** Access Token(30분)은 짧게, Refresh Token(14일)은 길게.
  - Access Token이 만료되면 Refresh Token으로 새 Access Token을 받는다.
  - 사용자는 14일 동안 자동 로그인 상태를 유지한다.
  - Refresh Token을 Redis에 저장해서 서버 측에서 강제 만료(로그아웃)시킬 수 있다.

**저장 위치:**
```
localStorage.accessToken   ← 30분짜리 JWT
localStorage.refreshToken  ← 14일짜리 JWT (Redis에도 저장)
localStorage.user          ← 사용자 기본정보 (쉼표번호, nickname, role 등)
```

**왜 localStorage인가:**
- `sessionStorage`는 탭을 닫으면 사라진다. 14일 자동로그인이 안 된다.
- `httpOnly cookie`가 더 안전하지만, 현재 프론트-백이 다른 포트(3000/8080)라서 CORS+cookie 설정이 복잡해진다. MVP 단계에서는 localStorage로 단순하게 처리했다.

---

### 4-2. 쉼표번호를 JWT의 subject로 쓴 이유

```java
// JwtUtil.java
Jwts.builder()
    .setSubject(쉼표번호)   // ← 여기에 담음
    .claim("role", role)
    .setExpiration(...)
    .signWith(key)
    .compact();
```

**왜:**
- JWT는 위변조가 불가능하다(서명 검증). 즉, subject에 있는 `쉼표번호`는 신뢰할 수 있다.
- 매 요청마다 DB를 조회하지 않아도 "이 요청이 누구 것인지" 알 수 있다.
- 인터셉터에서 `request.setAttribute("쉼표번호", 쉼표번호)`로 전달하면 Controller가 DB 조회 없이 사용자 ID를 바로 쓸 수 있다.

---

### 4-3. Redis를 Refresh Token 저장소로 쓴 이유

**왜:**
- **강제 로그아웃**이 가능하다. DB에 저장하면 삭제 시 DB 쿼리가 필요한데, Redis는 Key-Value로 즉시 삭제된다.
- **자동 만료**가 된다. Redis에 `TTL(14일)`을 설정하면 14일 후 자동으로 사라진다. DB면 배치 작업으로 만료 토큰을 정리해야 한다.
- 토큰 조회가 O(1)으로 빠르다. 매 요청마다 Refresh Token을 확인하는 건 아니지만, Redis가 훨씬 빠르다.
- 현재 Upstash Redis(클라우드)를 사용해서 서버 재시작해도 토큰이 유지된다.

---

### 4-4. 소셜 로그인 — Pending Token 방식

**흐름:**
```
카카오/구글/네이버 로그인 → 기존 계정이면 바로 JWT 발급
                         → 신규 가입이면 pending token 발급 (10분 유효)
                            → /kakao/confirm 페이지로 이동
                            → username 입력 후 계정 생성
                            → 정식 JWT 발급
```

**왜 중간 확인 단계가 있나:**
- 소셜 계정 정보만으로는 사용자가 원하는 로그인 아이디(username)를 알 수 없다. 직접 입력받아야 한다.
- 서버에 임시 세션을 만들지 않고 JWT 안에 소셜 정보를 담아 stateless를 유지한다.
- pending token은 `claim("pending", true)`로 표시해서 정식 access token과 구분한다.

## 5. 데이터베이스 설계 이유

### 5-1. 쉼표번호를 String PK로 쓴 이유

```java
// User.java
private String 쉼표번호;  // 예: "쉼표0001", "쉼표9001"
```

**왜:**
- 처음에 사용자에게 보여줄 닉네임(`쉼표0001`)을 PK로 쓰는 설계를 했다.
- 이 결정으로 34개 테이블이 모두 `쉼표번호`를 FK로 참조하고 있다.
- **이상적인 설계는 아니다.** `id BIGINT AUTO_INCREMENT`를 PK로 쓰고 `쉼표번호`는 별도 컬럼이어야 한다.
- 하지만 MVP 완성 후에 리팩토링 예정이고, 지금 바꾸면 34개 테이블의 FK를 전부 수정해야 해서 MVP 이후로 미뤘다.
- **절대로 BIGINT로 변환하면 안 된다.** 문자열이라 `쉼표0001` 같은 한글+숫자 포맷이 가능하다.

---

### 5-2. 원천 데이터 vs 산출물 분리

```
heart_rate_measurements  ← 원천 (BPM 측정값, 변경 불가)
survey_responses         ← 원천 (사용자 응답, 변경 불가)
↓ (계산)
diagnosis_results        ← 산출물 (언제든 재계산 가능)
rest_type_scores         ← 산출물 (진단결과에서 도출)
```

**왜:**
- 진단 알고리즘이 바뀌어도 원천 데이터(심박, 응답)가 있으면 **재계산**할 수 있다.
- 원천 데이터를 지우면 재계산이 불가능하다. 반드시 유지해야 한다.
- 통계 이상 발견 시 원천 데이터로 거슬러 올라가서 검증할 수 있다.
- 멘토링에서 "핵심 흐름의 데이터가 얕다"는 경고를 받은 이후, 이 구조를 확립했다.

---

### 5-3. recommendations 테이블을 반드시 저장하는 이유

```
recommendations
├── 쉼표번호 (누가)
├── 진단결과ID (어떤 진단 기반으로)
├── 장소ID (어떤 장소를)
├── 추천기준 (왜 추천했나)
├── 클릭여부 (클릭했나)
├── 저장여부 (북마크했나)
└── 추천일시 (언제)
```

**왜:**
- **추천 클릭률, 저장률**은 서비스의 핵심 성공 지표다.
- 이 데이터가 없으면 "추천 기능이 얼마나 잘 동작하는가"를 측정할 방법이 없다.
- 통계/개선 페이지가 실제 데이터 기반이 아닌 가짜 데이터를 보여주게 된다.
- "화면은 많은데 데이터가 얕다"는 멘토링 경고를 받은 핵심 원인 중 하나였다.

### 5-4. monthly_stats 집계 테이블을 따로 둔 이유

```sql
monthly_stats
├── 쉼표번호
├── 년월 (예: "2026-03")
├── 총휴식시간_분
├── 유형별비율JSON
├── 평균감정점수
└── 기록횟수
```

**왜:**
- `rest_logs`에서 매번 집계 쿼리를 돌리면 데이터가 많아질수록 느려진다.
- 미리 집계해두면 통계 페이지 조회가 단순한 SELECT 하나로 끝난다.
- 유형별 비율을 JSON으로 저장한 이유는 휴식 유형 7가지의 비율을 컬럼 7개로 늘리는 것보다 JSON 하나로 유연하게 관리하기 위해서다.
- 월 단위로 집계되므로, 기록이 추가될 때마다(rest_logs INSERT) 해당 월의 monthly_stats를 UPDATE한다.

### 5-5. soft delete를 쓴 이유 (rest_logs의 삭제여부 컬럼)

```sql
rest_logs.삭제여부  -- 0: 정상, 1: 삭제
```

**왜:**
- 실제로 DB에서 ROW를 지우면 복구가 불가능하다.
- 통계 집계 시 "삭제된 기록은 제외"하는 필터만 추가하면 된다.
- 나중에 "삭제한 기록 복원" 기능을 추가하기 쉽다.
- 사용자 행동 데이터는 분석에 가치가 있어서 완전 삭제보다 soft delete가 적합하다.

## 6. API 응답 형식 이유

### 6-1. ApiResponse 래퍼 클래스

```java
// 모든 API가 이 형식으로 응답
{ "success": true,  "data": { ... }, "message": "처리완료" }
{ "success": false, "data": null,    "message": "이메일이 이미 존재합니다" }
```

**왜:**
- HTTP 상태코드(200, 400, 401, 500)만으로는 비즈니스 레벨의 성공/실패를 구분하기 어렵다.
- 예: 로그인 시 이메일은 맞지만 비밀번호가 틀린 경우 → HTTP 400이지만 `message`로 정확한 이유 전달.
- 프론트엔드에서 `if (data.success)` 하나로 성공/실패를 판별할 수 있다.
- `message`는 UI에 바로 보여줄 수 있는 사용자 친화적인 메시지다.
- 모든 API가 동일한 형식이라서 프론트에서 공통 에러 처리를 만들 수 있다.

**ApiResponse.java 구조:**
```java
public class ApiResponse<T> {
    private final boolean success;
    private final T data;
    private final String message;

    public static <T> ApiResponse<T> ok(T data, String message) { ... }
    public static <T> ApiResponse<T> ok(String message) { ... }           // 데이터 없는 성공
    public static <T> ApiResponse<T> fail(String message) { ... }
}
```

제네릭 `<T>`를 써서 어떤 타입의 data든 감쌀 수 있게 했다.

## 7. 주요 기능별 설계 이유

### 7-1. 진단 시스템 — 심박 + 설문 분리

**흐름:**
```
measurement_sessions (심박 세션 시작)
  → heart_rate_measurements (BPM 데이터 저장)
  → survey_responses (설문 응답 저장)
  → diagnosis_results (종합 진단 결과)
  → rest_type_scores (유형별 점수)
```

**왜 이렇게 단계를 나눴나:**
- 심박 측정과 설문은 독립적으로 진행된다. 심박 없이 설문만 해도 진단이 가능해야 한다.
- `measurement_sessions`를 먼저 만들어서 세션 ID를 채번한 뒤, 그 세션에 측정값을 붙이는 방식이다.
- 세션 단위로 묶으면 "오늘 2번 진단했을 때" 각각을 분리해서 볼 수 있다.

**설문 점수 계산 방식:**
- 같은 유형의 질문이 여러 개인 경우 → **평균값** 사용
- 예: `physical` 유형 질문이 Q1(80점), Q13(60점) → 평균 70점
- Q8~Q12는 설계 불량으로 삭제됐고, 현재는 Q1~Q7, Q13~Q17 총 12문항이 사용된다.

### 7-2. HeartRateCheck — 동적 URL + QR코드

```
사용자 PC에서 측정 시작
  → 고유 세션 URL 생성 (예: /measure/abc123)
  → QR코드로 스마트폰에 공유
  → 스마트폰 브라우저에서 심박 측정
  → 측정값이 PC 화면에 실시간 반영
```

**왜:**
- 스마트폰의 카메라/센서로 심박을 측정하는 게 더 정확하다.
- PC에서 진단을 시작했지만 측정은 폰으로 하고 싶은 사용자를 위한 UX다.
- QR코드로 URL을 공유하면 URL을 손으로 입력하지 않아도 된다.
- 복사 버튼도 함께 제공해서 카카오톡 등으로도 공유 가능하다.

### 7-3. 추천 알고리즘 — 날씨·시간대 가중치 보정

**왜 진단 점수만으로 추천하지 않나:**
- 맥락 없는 추천은 신뢰도를 떨어뜨린다. 비 오는 날 "공원 산책"을 1순위로 추천하면 사용자는 앱을 믿지 않는다.
- 지금 당장 실행 가능한 추천을 하는 게 서비스의 핵심 가치다.

**가중치 예시:**
```java
// 비 오는 날 → 실내 활동 올리고, 야외 활동 내림
RAINY_BOOST = Map.of(
    "sensory",  +15,   // 카페·공방 (실내)
    "creative", +15,   // 실내 집중
    "nature",   -20,   // 공원·산 (야외) 억제
    "physical", -20    // 야외 운동 억제
);

// 저녁 시간
EVENING_BOOST = Map.of(
    "social",   +20,   // 저녁 약속
    "sensory",  +10    // 분위기 있는 카페
);
```

### 7-4. YOLO 이미지 분석 서비스 분리

```
Spring Boot (8080) ←HTTP→ YOLO FastAPI (8090, Python)
```

**왜 별도 서비스로:**
- YOLOv8은 Python 생태계의 라이브러리다. Java에서 직접 실행할 수 없다.
- Spring Boot에서 HTTP로 YOLO 서버를 호출하는 방식이 가장 단순하다.
- YOLO 서버가 꺼져 있어도 Spring Boot는 정상 동작한다 (graceful degradation). 이미지 분류만 안 될 뿐이다.
- 나중에 YOLO 모델을 교체하거나 업그레이드할 때 Spring Boot 코드를 건드리지 않아도 된다.

**두 가지 역할:**
1. 관리자가 장소를 승인할 때 → 첫 번째 사진을 YOLO에 보내서 휴식 유형 태그 자동 생성
2. 커뮤니티 사진 업로드 시 → 부적절한 이미지(무기류 등) 자동 거부

---

### 7-5. 장소 상태 관리 (pending → approved → rejected)

```sql
places.status  -- 'pending' | 'approved' | 'rejected'
```

**왜:**
- 누구나 장소를 등록할 수 있게 하되, 관리자가 검토 후 공개한다.
- `pending`: 등록됐지만 아직 검토 안 됨 → 일반 사용자에게 안 보임
- `approved`: 관리자가 승인 → 지도/추천에 표시, YOLO 자동 태그 실행
- `rejected`: 관리자가 거부 → 등록자에게 알림
- 이 구조 덕분에 스팸/부적절한 장소가 바로 노출되는 걸 막는다.

### 7-6. 휴식 유형 7가지 — Enum 대신 테이블로

```sql
rest_types 테이블
├── physical  (신체적 휴식)
├── mental    (정신적 휴식)
├── sensory   (감각적 휴식)
├── emotional (감정적 휴식)
├── social    (사회적 휴식)
├── creative  (창의적 휴식)
└── nature    (자연 속 휴식)
```

**왜 테이블인가:**
- Enum(Java enum 또는 DB enum)은 값을 추가/수정하려면 코드/스키마를 수정해야 한다.
- 테이블로 관리하면 관리자 화면에서 이름, 설명, 아이콘, 색상을 DB에서 바로 수정할 수 있다.
- 향후 휴식 유형을 추가해도 코드 수정 없이 INSERT 하나로 끝난다.
- Seed 데이터(7개)가 반드시 있어야 화면이 정상 동작한다.

### 7-7. 금칙어 소프트 필터링 (shadow hide)

**방식:**
```
게시글/댓글 작성
  → blocked_keywords 테이블과 비교
  → 금칙어 없으면 → status = 'visible' (정상 공개)
  → 금칙어 있으면 → status = 'hidden' (본인만 볼 수 있음)
```

**왜 거부하지 않고 숨기나:**
- 거부 메시지를 보내면 사용자가 어떤 단어가 금칙어인지 역으로 파악할 수 있다.
- 숨김 처리하면 본인은 게시됐다고 생각하지만 다른 사람에게는 안 보인다.
- DB에 남아있으므로 관리자가 나중에 검토해 복원하거나 완전 삭제할 수 있다.
- 외부 유료 필터링 서비스 없이 `blocked_keywords` 테이블만으로 운영 가능하다.

### 7-8. analytics_events — @Async 비동기 수집

**왜:**
- 이벤트 DB 저장이 실패하거나 느려도 사용자의 실제 요청(기록 저장, 진단 완료)은 영향받지 않아야 한다.
- `@Async`를 붙이면 `analyticsService.track(...)` 호출 즉시 반환되고, 실제 저장은 별도 스레드에서 처리된다.
- `try/catch`로 실패를 조용히 무시한다. 수집 실패가 서비스 장애로 이어지면 안 된다.

### 7-9. 커뮤니티/챌린지를 핵심 흐름 완성 후로 미룬 이유

**왜:**
- 멘토링에서 "핵심 흐름(진단→추천→기록→개선)의 데이터가 얕다"는 경고를 받았다.
- 커뮤니티는 있으면 좋지만 서비스의 핵심 가치(휴식 추천/기록)와 직접 연관이 없다.
- 제한된 시간 안에 핵심 흐름을 완성하는 게 우선이다.
- **결과**: 핵심 흐름 완성 후 커뮤니티, 챌린지, 신고, 금칙어, analytics까지 모두 구현 완료.

### 7-10. 챌린지/배지 시스템 설계 이유

**테이블 구조:**
```
challenges              ← 챌린지 정의 (제목, 조건, 기간, 보상 배지)
  → user_challenges     ← 사용자 참여 기록 (진행률, 완료 여부)
  → challenge_verifications ← 매일 인증 기록
badges                  ← 배지 정의
  → user_badges         ← 사용자가 획득한 배지
```

**왜 챌린지를 만들었나:**
- 휴식 기록은 단발성으로 끝나기 쉽다. "7일 연속 기록" 같은 미션이 있어야 지속적인 사용이 생긴다.
- 게임화(gamification)로 서비스 리텐션을 높이는 검증된 방법이다.

**배지를 별도 테이블로 분리한 이유:**
- 챌린지 달성 외에도 "첫 심박 측정", "10회 기록 달성" 같은 다른 조건으로도 배지를 줄 수 있어야 한다.
- `badges` 테이블에 조건을 정의하고, `user_badges`에 획득 이력을 쌓으면 조건 추가 시 코드 수정 없이 INSERT 하나로 끝난다.

**진행률 계산 방식:**
- `user_challenges.progress` = 인증 횟수 / 목표 횟수 × 100
- 챌린지가 완료되면 `UserChallengeService`에서 자동으로 배지 획득 처리(`user_badges` INSERT)한다.
- 챌린지 완료와 배지 획득은 `@Transactional` 하나로 묶어서 배지가 누락되는 경우를 방지했다.

**공개 목록 vs 참여 후:**
- 챌린지 목록(`/api/challenges`)은 비로그인도 볼 수 있다 — `JwtInterceptor` PUBLIC_PATHS에 포함.
- 참여·인증·배지 조회는 로그인 필수.

### 7-11. 커뮤니티 시스템 설계 이유

**테이블 구조:**
```
posts         ← 게시글 (익명 여부, status: visible/hidden)
  → comments  ← 댓글 (익명 지원)
  → post_likes   ← 좋아요 (쉼표번호 + post_id UNIQUE)
  → post_reports ← 신고 (신고 사유, 처리 상태)
blocked_keywords ← 금칙어 목록 (관리자 관리)
```

**익명 게시글을 지원한 이유:**
- 휴식/스트레스 관련 솔직한 경험은 실명으로 쓰기 부담스럽다.
- `is_anonymous` 컬럼으로 익명 여부를 저장하되, DB에는 쉼표번호가 남아있어 관리자가 조회 가능하다.
- 완전 익명이 아니라 "운영자에게는 보이는 익명"이라서 악용 시 제재가 가능하다.

**좋아요에 UNIQUE 제약을 건 이유:**
- `(쉼표번호, post_id)`에 UNIQUE를 걸면 중복 좋아요가 DB 레벨에서 차단된다.
- 서비스 레이어에서 별도 중복 체크 없이 INSERT 실패로 자동 처리된다.

**신고 처리 흐름:**
```
사용자 신고 → post_reports INSERT (status: pending)
  → 관리자 화면에 노출
  → 관리자 처리 → status: resolved / dismissed
  → 처리 시 해당 게시글 status 변경 가능 (visible → hidden)
```

**금칙어 shadow-hide 방식:**
- 거부(block)하면 사용자가 어떤 단어가 금칙어인지 역추적할 수 있다.
- 숨김(hidden) 처리하면 작성자는 게시됐다고 생각하지만 다른 사람에게는 보이지 않는다.
- DB에 남아있으므로 관리자가 복원하거나 완전 삭제 가능하다.

## 8. 개발 환경 구성 이유

### 8-1. docker-compose.yml로 로컬 MySQL + Redis를 띄운 이유

```yaml
services:
  mysql:
    image: mysql:8.0
    ports: ["3306:3306"]
  redis:
    image: redis:7
    ports: ["6379:6379"]
```

**왜:**
- 팀원마다 MySQL을 직접 설치하면 버전이 달라서 문제가 생긴다.
- Docker로 띄우면 `docker-compose up -d` 하나로 동일한 환경이 된다.
- 로컬 DB를 건드려도 클라우드 DB(Cloud SQL)에 영향이 없다.
- 로컬에서 개발/테스트 후 클라우드에 배포하는 workflow가 된다.

---

### 8-2. application.yml / application-local.yml / application-prod.yml 분리

```
application.yml       ← 공통 설정 (포트, MyBatis 설정 등)
application-local.yml ← 로컬용 (gitignore에 추가, 실제 비밀번호 포함)
application-prod.yml  ← 배포용 (환경변수 참조)
```

**왜:**
- 비밀번호, API 키를 git에 올리면 안 된다. `application-local.yml`은 `.gitignore`에 포함.
- 로컬 DB 주소와 클라우드 DB 주소가 다르다. 분리하면 스위칭이 쉽다.
- 배포 환경에서는 환경변수(`${DB_PASSWORD}`)로 주입받아서 이미지에 비밀값이 포함되지 않는다.

---

### 8-3. GlobalExceptionHandler를 만든 이유

```java
// global/exception/GlobalExceptionHandler.java
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(Exception.class)
    public ApiResponse<?> handleException(Exception e) {
        return ApiResponse.fail(e.getMessage());
    }
}
```

**왜:**
- 예외가 발생했을 때 각 Controller에서 try/catch로 잡지 않아도 된다.
- 처리되지 않은 예외가 `500 Internal Server Error` HTML 페이지로 반환되는 걸 막는다.
- 모든 에러가 `ApiResponse.fail(message)` 형식으로 통일되어 프론트에서 일관되게 처리할 수 있다.

---

### 8-4. PlaceScoreScheduler — 스케줄러로 장소 점수 자동 갱신

```java
// global/scheduler/PlaceScoreScheduler.java
@Scheduled(cron = "0 0 3 * * *")  // 매일 새벽 3시
public void updatePlaceScores() {
    // 리뷰 별점, 북마크 수, 추천 클릭률 기반으로 AI 점수 재계산
}
```

**왜:**
- 장소 점수는 리뷰/북마크가 쌓일수록 변한다. 실시간으로 계산하면 API가 느려진다.
- 하루에 한 번 새벽에 배치로 계산해두면 조회 시 미리 계산된 값을 그냥 읽기만 하면 된다.
- 새벽 3시를 선택한 이유는 트래픽이 가장 적은 시간대이기 때문이다.

---

### 8-5. ClaudeApiClient를 별도 클래스로 분리한 이유

```java
// global/config/ClaudeApiClient.java
public class ClaudeApiClient {
    // Claude API 호출 래퍼
    // messages API, system prompt 설정
    // 응답 파싱 (JSON → String)
}
```

**왜:**
- Claude API 호출 로직이 여러 Service에서 쓰인다 (추천 이유 생성, 휴식 가이드 등).
- 매 Service마다 HTTP 요청 코드를 반복하지 않고 `ClaudeApiClient.chat(prompt)`로 단순하게 쓴다.
- API 키 관리, 요청/응답 형식을 한 곳에서 관리한다.
- Claude API 버전이 바뀌어도 이 파일 하나만 수정하면 된다.

## 결론 — 설계의 핵심 원칙

이 프로젝트의 모든 설계 결정은 세 가지 원칙을 따랐다:

### 1. MVP 완성 우선
> "화면은 많은데 데이터가 얕다"는 멘토링 경고에 따라, 핵심 흐름(진단→추천→기록→개선)이 실제 데이터로 동작하는 것을 최우선으로 했다.

### 2. 단순하고 이해 가능한 코드
> Spring Security 대신 JwtInterceptor, JPA 대신 MyBatis + SQL, axios 대신 fetch. 복잡한 추상화보다 팀원 모두가 이해할 수 있는 직관적인 코드를 선택했다.

### 3. 확장 가능한 구조
> 도메인 패키지 구조, 공통 컴포넌트, ApiResponse 표준, 원천 데이터 보존 등은 지금 당장 필요해서가 아니라 나중에 기능을 추가하거나 수정할 때 최소한의 변경으로 대응하기 위한 투자다.

---

*이 문서는 코드를 처음 보는 사람, 혹은 나중에 다시 이 코드를 보는 나 자신을 위해 작성했다.*
*설계는 완벽하지 않다. MVP 이후 리팩토링이 예정된 부분(쉼표번호 PK, localStorage 인증 등)도 솔직하게 담았다.*
