## 1. 전체 구조 한눈에 보기

```
사용자 브라우저 (React, localhost:3000)
        ↓ HTTP 요청 (fetch)
┌─────────────────────────────────────┐
│  Spring Boot (localhost:8080)       │
│                                     │
│  JwtInterceptor ← 모든 요청 먼저 통과 │
│       ↓                             │
│  Controller → Service → Mapper      │
│                   ↓                 │
│             MySQL (Cloud SQL)       │
│             Redis (토큰 저장)        │
└─────────────────────────────────────┘
        ↓ 필요할 때만
  YOLO FastAPI (localhost:8090)
  이미지 분석 전용 Python 서버
```

**핵심 흐름:**
```
회원가입/로그인 → JWT 발급 → 심박+설문 진단 → 장소 추천 → 휴식 기록 → 통계
```

이 흐름 하나가 서비스의 전부다. 나머지 커뮤니티, 챌린지, 관리자는 이 흐름을 보조한다.

### ① `JwtInterceptor.java` — 모든 API 요청의 문지기

**경로:** `backend/src/main/java/com/comma/global/config/JwtInterceptor.java`

**한 줄 요약:** 로그인이 필요한 API와 아닌 API를 구분하고, JWT 토큰이 유효한지 검사한다.

**왜 중요한가:**
- Spring Security 없이 인증을 직접 구현한 핵심 파일이다.
- 이 파일 하나가 프로젝트 전체의 보안을 담당한다.
- 코드가 100줄 미만으로 짧아서 전체 흐름을 한눈에 볼 수 있다.

**코드와 설명:**

```java
// 이 경로들은 로그인 없이 접근 가능
private static final String[] PUBLIC_PATHS = {
    "/api/auth/signup",    // 회원가입
    "/api/auth/login",     // 로그인
    "/api/places",         // 장소 목록 (비로그인도 탐색 가능)
    "/api/posts",          // 커뮤니티 글 목록
    "/api/challenges",     // 챌린지 목록
    // ...
};
```

```java
@Override
public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {

    // 1단계: CORS 사전 요청(OPTIONS)은 무조건 통과
    if ("OPTIONS".equalsIgnoreCase(request.getMethod())) return true;

    String path = request.getServletPath();

    // 2단계: 공개 경로인지 확인
    boolean isPublic = false;
    for (String pattern : PUBLIC_PATHS) {
        if (pathMatcher.match(pattern, path)) { isPublic = true; break; }
    }

    if (isPublic) {
        // 공개 경로여도 토큰이 있으면 사용자 정보 추출 (선택적 인증)
        // → "내가 북마크한 장소인가?" 같은 기능에 활용
        String authHeader = request.getHeader("Authorization");
        if (authHeader != null && authHeader.startsWith("Bearer ")) {
            String token = authHeader.substring(7);
            if (jwtUtil.isTokenValid(token)) {
                request.setAttribute("쉼표번호", jwtUtil.extract쉼표번호(token));
            }
        }
        return true; // 통과
    }

    // 3단계: 로그인 필요한 경로 — 토큰 없으면 401
    String header = request.getHeader("Authorization");
    if (header == null || !header.startsWith("Bearer ")) {
        sendError(response, 401, "로그인이 필요합니다.");
        return false; // 컨트롤러 실행 안 됨
    }

    // 4단계: 토큰 유효성 검사
    String token = header.substring(7);
    if (!jwtUtil.isTokenValid(token)) {
        sendError(response, 401, "유효하지 않은 토큰입니다. 다시 로그인해주세요.");
        return false;
    }

    String 쉼표번호 = jwtUtil.extract쉼표번호(token);
    String role = jwtUtil.extractRole(token);

    // 5단계: /api/admin/** 경로는 ADMIN만 허용
    if (pathMatcher.match("/api/admin/**", path) && !"ADMIN".equals(role)) {
        sendError(response, 403, "관리자 권한이 필요합니다.");
        return false;
    }

    // 검증 완료 → 컨트롤러에서 꺼내 쓸 수 있도록 저장
    request.setAttribute("쉼표번호", 쉼표번호);
    request.setAttribute("role", role);
    return true; // 통과. 컨트롤러 실행
}
```

**리뷰 포인트 (잘 된 것):**
- `return true` / `return false` 만으로 통과/차단을 결정해서 흐름이 단순하다.
- `request.setAttribute("쉼표번호", ...)` 패턴으로 컨트롤러까지 사용자 ID를 깔끔하게 전달한다.
- "공개 경로여도 토큰이 있으면 사용자 정보를 꺼내두는" 선택적 인증이 영리하다.

**설명할 때 이렇게 말해:**
> "Spring Security를 쓰지 않고 HandlerInterceptor를 직접 구현했습니다.
> 모든 API 요청이 컨트롤러에 도달하기 전에 이 인터셉터를 통과합니다.
> PUBLIC_PATHS 배열로 인증이 필요 없는 경로를 명시적으로 관리했고,
> 나머지는 JWT 토큰을 검증해서 유효하면 통과, 없거나 만료됐으면 401을 반환합니다."

### ② `AuthService.java` — 회원가입과 로그인 로직

**경로:** `backend/src/main/java/com/comma/domain/auth/service/AuthService.java`

**한 줄 요약:** 회원가입, 로그인, 토큰 갱신, 이메일 인증, 비밀번호 재설정을 처리한다.

**왜 중요한가:**
- 서비스의 시작점. 회원가입이 안 되면 아무것도 안 된다.
- 보안에 가장 민감한 코드다. BCrypt 암호화, JWT 발급이 여기서 일어난다.

**회원가입 핵심 코드:**

```java
@Transactional  // 중간에 실패하면 insertUser + insertEmailVerification 둘 다 취소
public User signup(String email, String username, String password) {

    // 중복 체크
    if (authMapper.countByEmail(email.trim()) > 0)
        throw new IllegalArgumentException("이미 사용 중인 이메일입니다.");
    if (authMapper.countByUsername(username.trim()) > 0)
        throw new IllegalArgumentException("이미 사용 중인 아이디입니다.");

    String 쉼표번호 = generate쉼표번호(); // DB에서 마지막 번호 조회 후 +1

    User user = new User();
    user.set쉼표번호(쉼표번호);
    user.setPassword(BCrypt.hashpw(password, BCrypt.gensalt())); // 단방향 암호화
    user.setNickname(쉼표번호);   // 처음엔 쉼표번호 = 닉네임. 나중에 변경 가능
    user.setRole("USER");

    authMapper.insertUser(user);

    // 개발 환경(app.skip-email-verification=true)이면 인증 스킵
    if (skipEmailVerification) {
        authMapper.updateUserEmailVerified(쉼표번호);
    } else {
        String token = UUID.randomUUID().toString(); // 이메일 인증 링크용 토큰
        mailService.sendVerificationEmail(email, username, token);
    }
    return user;
}
```

**로그인 핵심 코드:**

```java
public Map<String, Object> login(String email, String password) {

    User user = authMapper.findByEmail(email.trim());
    if (user == null)
        throw new IllegalArgumentException("이메일 또는 비밀번호가 올바르지 않습니다.");
        // ↑ "이메일이 없습니다"가 아닌 같은 메시지 → 보안상 이메일 존재 여부를 숨김

    if (!BCrypt.checkpw(password, user.getPassword())) // 입력값 vs 해시값 비교
        throw new IllegalArgumentException("이메일 또는 비밀번호가 올바르지 않습니다.");

    String accessToken  = jwtUtil.generateAccessToken(user.get쉼표번호(), user.getRole());  // 30분
    String refreshToken = jwtUtil.generateRefreshToken(user.get쉼표번호());                 // 14일

    // Refresh Token을 Redis에 저장 (TTL 14일, 서버에서 강제 만료 가능)
    redisTemplate.opsForValue().set("RT:" + user.get쉼표번호(), refreshToken, 14, TimeUnit.DAYS);

    user.setPassword(null); // 응답에 비밀번호 절대 포함 금지

    return Map.of("accessToken", accessToken, "refreshToken", refreshToken, "user", user);
}
```

**리뷰 포인트 (잘 된 것):**
- 로그인 실패 메시지를 "이메일 또는 비밀번호" 하나로 통일 → 이메일 존재 여부 노출 방지 (보안 기법)
- `user.setPassword(null)` → 응답에 해시값이 포함되지 않음
- `@Transactional`로 회원가입 중 메일 발송 실패해도 사용자 데이터는 살아있음

**설명할 때 이렇게 말해:**
> "로그인 실패 시 '이메일이 틀렸습니다', '비밀번호가 틀렸습니다'를 구분하지 않고
> 같은 메시지를 반환합니다. 공격자가 이메일 존재 여부를 파악하지 못하도록 하는
> 보안 관행입니다. 비밀번호는 BCrypt로 단방향 해시 처리해서 DB에 원본이 없습니다."

### ④ `ApiResponse.java` — 모든 API 응답의 공통 형식

**경로:** `backend/src/main/java/com/comma/global/util/ApiResponse.java`

**한 줄 요약:** 모든 API가 `{ success, data, message }` 형식으로 응답하도록 강제하는 클래스.

**왜 중요한가:**
- 짧지만(30줄) 프로젝트 전체에 영향을 주는 파일이다.
- 프론트엔드에서 모든 API를 동일한 방식으로 처리할 수 있게 해준다.

**전체 코드:**

```java
@Getter
public class ApiResponse<T> {
    private final boolean success;  // 성공 여부
    private final T data;           // 실제 데이터 (<T>: 어떤 타입이든 담을 수 있음)
    private final String message;   // 사용자에게 보여줄 메시지

    private ApiResponse(boolean success, T data, String message) {
        this.success = success;
        this.data = data;
        this.message = message;
    }

    // 사용법 ① 성공 + 데이터
    public static <T> ApiResponse<T> ok(T data, String message) {
        return new ApiResponse<>(true, data, message);
    }

    // 사용법 ② 성공인데 줄 데이터가 없을 때 (삭제 성공 등)
    public static <T> ApiResponse<T> ok(String message) {
        return new ApiResponse<>(true, null, message);
    }

    // 사용법 ③ 실패
    public static <T> ApiResponse<T> fail(String message) {
        return new ApiResponse<>(false, null, message);
    }
}
```

**프론트에서 사용 예시:**

```javascript
// 모든 API 응답이 같은 구조라서 공통 처리가 가능
const data = await fetchWithAuth('/api/places');
if (data.success) {
    setPlaces(data.data);
} else {
    alert(data.message); // 서버에서 온 메시지 바로 사용
}
```

**리뷰 포인트 (잘 된 것):**
- `private` 생성자 + `static` 팩토리 메서드 — 잘못된 방식으로 객체를 만들 수 없다.
- `<T>` 제네릭 — User, List\<Place\>, Map 등 어떤 타입도 `data`에 담을 수 있다.
- 프론트가 `if (data.success)`만으로 성공/실패를 판별한다 — 일관성.

**설명할 때 이렇게 말해:**
> "프로젝트 전체에서 API 응답 형식을 ApiResponse 클래스 하나로 통일했습니다.
> 모든 응답이 success, data, message 세 개의 필드를 가집니다.
> 덕분에 프론트엔드에서 if (data.success) 하나로 성공/실패를 처리하고,
> 실패 메시지는 data.message를 바로 화면에 보여주면 됩니다."

### ⑤ `PostService.java` — 커뮤니티 핵심 로직

**경로:** `backend/src/main/java/com/comma/domain/community/service/PostService.java`

**한 줄 요약:** 게시글 작성, 금칙어 필터링(shadow-hide), 댓글, 좋아요, 신고를 처리한다.

**왜 중요한가:**
- 커뮤니티의 핵심 보안 로직(금칙어 처리)이 여기 있다.
- 단순 CRUD가 아닌 shadow-hide 방식이 이 서비스의 특징이다.

**금칙어 처리 핵심 코드:**

```java
@Transactional
public Post createPost(String 쉼표번호, String title, String content, ...) {

    // 금칙어 검사 — 제목과 본문 합쳐서 검사

    post.setStatus(hasBlocked ? "hidden" : "visible");

    communityMapper.insertPost(post);

    // 작성자에게는 201 Created 성공 응답을 보냄 → 차단됐는지 모름
    return post;
}
```

```java
public boolean toggleLike(String 쉼표번호, Long postId) {

    if (alreadyLiked) {
        communityMapper.deleteLike(쉼표번호, postId);
        communityMapper.decrementLikeCount(postId);
        return false; // 좋아요 취소

        return true;  // 좋아요 추가
    }
}
```

### ⑥ `fetchWithAuth.js` — 인증 자동 처리 유틸

**경로:** `frontend/src/api/fetchWithAuth.js`

**한 줄 요약:** JWT를 자동으로 붙여서 API를 호출하고, 토큰이 만료되면 자동으로 갱신해서 재시도한다.

**왜 중요한가:**
- 프로젝트 전체 API 호출이 이 함수를 통해 이루어진다.
- 토큰 자동 갱신 로직이 없으면 30분마다 강제 로그아웃이 발생한다.

**핵심 코드와 설명:**

```javascript
let isRefreshing = false;  // 지금 토큰 갱신 중인가?
let refreshQueue = [];     // 갱신 중에 들어온 다른 요청들 대기

export async function fetchWithAuth(url, options = {}) {
  const accessToken = localStorage.getItem('accessToken');

  const headers = {
    'Content-Type': 'application/json',
    ...options.headers,
  };

  if (accessToken) {
    headers['Authorization'] = `Bearer ${accessToken}`; // 토큰 자동 첨부
  }

  let res = await fetch(url, { ...options, headers });

  // 401 = 토큰 만료 → 자동 갱신 시도
  if (res.status === 401 && localStorage.getItem('refreshToken')) {
    try {
      let newToken;

      if (isRefreshing) {
        // 이미 갱신 중이면 큐에 등록하고 기다림
        // 여러 요청이 동시에 401을 받아도 갱신은 딱 1번만 실행됨
        newToken = await new Promise((resolve, reject) => {
          refreshQueue.push({ resolve, reject });
        });
      } else {
        isRefreshing = true;
        try {
          newToken = await refreshAccessToken(); // 새 토큰 발급
          processRefreshQueue(newToken);          // 대기 중이던 요청들도 재시도
        } catch {
          forceLogout(); // 갱신도 실패하면 강제 로그아웃
          throw refreshError;
        } finally {
          isRefreshing = false;
        }
      }

      // 새 토큰으로 원래 요청 재시도
      headers['Authorization'] = `Bearer ${newToken}`;
      res = await fetch(url, { ...options, headers });
    } catch {
      forceLogout();
    }
  }

  return res.json();
}
```

**리뷰 포인트 (잘 된 것):**
- `isRefreshing` 플래그 + `refreshQueue` 배열로 동시 요청 처리 — 토큰 갱신이 N번 실행되는 문제를 막았다.
- 갱신 실패 시 자동으로 `forceLogout()` 호출 — 사용자가 무한 루프에 빠지지 않는다.

**설명할 때 이렇게 말해:**
> "Access Token이 30분이라서 사용 중에 만료될 수 있습니다.
> fetchWithAuth는 401 응답을 받으면 자동으로 Refresh Token으로 새 Access Token을 받아서
> 원래 요청을 재시도합니다. 사용자 입장에서는 중단 없이 서비스가 계속됩니다.
> 동시에 여러 요청이 401을 받아도 토큰 갱신은 1번만 실행되도록 처리했습니다."

## 5. 솔직한 개선 가능 부분

코드 리뷰에서 이 부분들이 지적될 수 있다. 미리 알고 있으면 당황하지 않는다.

### ⚠️ 쉼표번호 String PK

```java
private String 쉼표번호; // 예: "쉼표0001"
```

**문제:** PK가 String이라서 JOIN 쿼리가 느리다. 이상적으로는 `id BIGINT AUTO_INCREMENT`를 PK로 쓰고 `쉼표번호`는 별도 컬럼이어야 한다.

**왜 그대로 뒀나:** 27개 테이블이 이미 이 PK를 FK로 참조하고 있다. 지금 바꾸면 전체 스키마를 재설계해야 한다. MVP 완성 후 리팩토링 예정.

**말할 때:** "설계 초기의 결정이 나중에 제약이 된 사례입니다. MVP 이후 개선 계획이 있습니다."

---

### ⚠️ localStorage 토큰 저장

```javascript
localStorage.setItem('accessToken', token);
```

**문제:** XSS 공격에 취약하다. 더 안전한 방식은 httpOnly 쿠키다.

**왜 그대로 뒀나:** 프론트(3000)와 백엔드(8080)가 다른 포트라서 쿠키 설정이 복잡하다. MVP 단계에서는 단순하게 처리했다.

**말할 때:** "프로덕션에서는 httpOnly 쿠키로 마이그레이션이 필요합니다. 현재는 개발/데모 수준으로 localStorage를 사용했습니다."

---

### ⚠️ 한글 변수명 (쉼표번호)

```java
private String 쉼표번호;
request.getAttribute("쉼표번호");
```

**문제:** 한글 변수명은 JVM 환경에 따라 인코딩 이슈가 발생할 수 있다. MyBatis OGNL에서도 한글 파라미터명은 `@Param`으로 별도 명시가 필요하다.

**왜 그대로 뒀나:** 초기에 도메인 언어로 한글을 선택했고, 현재 잘 동작하고 있다. 다만 협업·이식성 측면에서 영문(`commaNo`)이 낫다.

**말할 때:** "도메인 용어를 코드에 직접 표현하려는 의도였습니다. 실제 운영 코드에서는 영문 변수명이 더 안전합니다."

---

### ⚠️ 테스트 코드 없음

**문제:** 단위 테스트, 통합 테스트가 없다. 기능 수정 시 다른 기능이 깨지는지 확인하기 어렵다.

**왜 그대로 뒀나:** 개인 프로젝트에서 개발 속도를 우선했다.

**말할 때:** "프로덕션 수준에서는 핵심 Service 로직에 대한 단위 테스트가 필요합니다. 다음 단계에서 추가할 계획입니다."

## 6. 코드 리뷰 시 자주 나오는 질문 & 답변

---

**Q. Spring Security를 왜 안 썼어요?**

A. Spring Security는 강력하지만 설정이 복잡합니다. SecurityFilterChain, UserDetailsService, GrantedAuthority 등 개념을 다 이해하고 설정해야 합니다. 이 프로젝트의 인증 요구사항은 "JWT가 있으면 통과, 없으면 401, ADMIN이면 관리자 API 허용"이 전부입니다. HandlerInterceptor 하나로 충분히 구현됐고, 코드가 단순해서 흐름을 바로 이해할 수 있습니다.

---

**Q. JPA 대신 MyBatis를 쓴 이유가 뭐예요?**

A. 이 프로젝트는 복잡한 집계 쿼리가 많습니다. 월간 통계, 진단 점수 계산, 날씨 보정 추천 등이 GROUP BY, CASE WHEN, 서브쿼리를 포함합니다. JPA는 단순 CRUD엔 편하지만 이런 복잡한 쿼리는 `@Query`로 직접 SQL을 쓰거나 QueryDSL을 추가해야 해서 오히려 복잡해집니다. MyBatis는 SQL을 그대로 쓰기 때문에 쿼리 최적화도 직접 할 수 있고, XML 파일만 보면 어떤 SQL이 실행되는지 바로 알 수 있습니다.

---

**Q. axios 대신 fetch를 쓴 이유가 있나요?**

A. fetch는 브라우저 내장 API라 별도 설치가 없습니다. `fetchWithAuth`라는 wrapper를 만들어서 JWT 자동 첨부, 401 시 토큰 자동 갱신 로직을 한 곳에 모았습니다. axios의 interceptor와 같은 역할을 직접 구현한 것입니다.

---

**Q. 금칙어 필터링이 차단이 아닌 숨김인 이유가 뭔가요?**

A. 차단 메시지를 보내면 사용자가 어떤 단어가 금칙어인지 계속 시도해서 알아낼 수 있습니다. 숨김 처리하면 작성자는 게시됐다고 생각하지만 다른 사람에게는 보이지 않습니다. 이를 shadow-hide 또는 shadow-ban이라고 하며, 실제 커뮤니티 서비스에서 많이 쓰는 방식입니다.

---

**Q. Redis를 왜 썼어요?**

A. Refresh Token 저장에 사용했습니다. Redis는 Key-Value 저장소로 TTL(만료 시간)을 설정할 수 있어서 14일 후 자동으로 사라집니다. MySQL에 저장하면 만료된 토큰을 배치로 삭제해야 하는데, Redis는 자동으로 처리됩니다. 또 강제 로그아웃 기능(서버에서 Refresh Token 삭제)도 Redis가 MySQL보다 빠릅니다.

---

**Q. 추천 알고리즘이 실제로 개인화되어 있나요?**

A. 네. 5가지 요소를 순차적으로 보정합니다. ① 휴식유형 진단 점수(0~100)가 기반이고, ② 심박수 stressIndex와 PSS-10 점수를 결합한 스트레스 레벨로 회복형/활동형 유형을 조정합니다. ③ 날씨(비오면 야외 억제), ④ 시간대(저녁엔 사교 강화), ⑤ 개인 이력·협업 필터링·패턴 학습 보정이 순서대로 누적됩니다. 추천 결과는 DB에 저장해서 나중에 "어떤 장소가 실제로 많이 선택됐는가"를 분석할 수 있는 구조입니다.

*모르는 것은 "현재 MVP 단계에서는 이렇게 구현했고, 개선 방향은 알고 있습니다"로 답하면 된다.*

## 가장 어려웠던 것

**DB 설계가 가장 힘들었다.**

처음에는 테이블을 단순하게 잡았다. 그런데 기능이 하나씩 붙을수록 FK가 꼬이고, 어디다 데이터를 저장해야 할지 계속 헷갈렸다. 심박수 측정값, 설문 응답, 진단 결과를 하나의 테이블에 다 넣으려다 실패했고, 결국 "원천 데이터와 산출물을 분리하라"는 피드백을 받고 나서야 구조가 잡혔다.

쉼표번호를 PK로 쓴 것도 초반의 실수였다. 그 결정 하나가 27개 테이블 전체에 영향을 줬고, 나중에 고치려 해도 건드릴 수가 없었다. 설계는 처음이 가장 중요하다는 걸 몸으로 배웠다.

---

## 예상보다 잘 된 것

**혼자인데 생각보다 많이 만들었다.**

처음에 목표로 잡은 핵심 흐름(진단 → 추천 → 기록)은 다 완성했고, 거기서 멈추지 않고 커뮤니티, 챌린지, 배지, 관리자 페이지, 소셜 로그인, 이메일 인증, 금칙어 필터, YOLO 이미지 분석까지 붙였다. 백엔드 API만 60개가 넘고 페이지는 80개가 넘는다. 처음에는 이게 가능할지 몰랐다.

심박수 측정 기능이 예상보다 잘 작동해서 좋았다. 스마트폰 카메라로 심박수를 재는 게 실제로 되는 게 신기했고, QR코드로 PC와 폰을 연결해서 데이터를 주고받는 부분이 개인적으로 가장 뿌듯했다.

---

## 배운 것

**기술적으로:**
- Spring Boot의 전체 흐름(Controller → Service → Mapper → DB)을 혼자 설계하고 구현해봤다.
- JWT 인증이 어떻게 동작하는지, 토큰을 어디에 저장하고 어떻게 검증하는지 실제로 짜봤다.
- 프론트와 백엔드가 서로 다른 포트에서 통신할 때 CORS 문제가 왜 생기는지, 어떻게 해결하는지 알게 됐다.
- React에서 데이터 로딩, 에러 처리, 로그인 상태에 따른 조건부 렌더링이 자연스러워졌다.

**개발 방식에 대해:**
- 한 번에 완성하려 하지 말고, 작동하는 것부터 완성하고 나서 붙여가는 게 맞다는 것.
- 화면을 많이 만드는 것보다 핵심 흐름의 데이터가 제대로 쌓이는 게 더 중요하다는 것.
- 설계 문서를 코드보다 먼저 쓰는 습관이 나중에 얼마나 도움이 되는지.

---

## 아쉬운 것

배포를 못 마친 게 아쉽다. Google Cloud Run 설정까지 다 잡아뒀는데, Docker 이미지 빌드와 Cloud SQL 연결 부분에서 막혀서 시간 안에 마무리하지 못했다. 기능은 다 완성됐는데 실제 URL로 보여주지 못하는 게 발표 때 가장 아쉬운 부분이다.

테스트 코드를 하나도 못 쓴 것도 아쉽다. 기능을 하나 고칠 때마다 다른 기능이 깨지지 않았는지 직접 눌러보며 확인했는데, 테스트가 있었다면 훨씬 빠르게 작업할 수 있었을 것 같다.

---

## 마지막으로

처음 이 프로젝트를 시작할 때는 백엔드도, 프론트도 제대로 만들어본 적이 없었다. 기획부터 설계, 개발, 문서화까지 전부 혼자 해야 한다는 게 막막했다.

그래도 끝까지 만들고 나니, 처음에 그렸던 흐름이 실제로 동작하는 서비스가 됐다. 완벽하진 않지만, 내가 상상한 것이 코드로 살아있다는 게 신기하고 기분이 좋다.

다음에 또 만든다면, 설계를 더 오래 고민하고, 테스트를 먼저 쓰고, 배포까지 마무리할 것이다.
