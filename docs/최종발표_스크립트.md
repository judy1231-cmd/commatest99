# 쉼표(,) 최종발표 스크립트

발표자: 강민정 (개인 프로젝트)
발표 시간: 약 20분
청중: 선생님 + 같은 반 학생들
베이스: 중간발표 스크립트1 기준으로 재작성
---

# [PART 1] 오프닝 & 프로젝트 소개 (약 1분 30초)

(슬라이드: 타이틀 화면)

안녕하세요 저는 이번에 개인으로 프로젝트 하게 된 강민정입니다.
먼저 말씀드리고 싶은 건 저는 이런 발표가 처음이라서 실수 할 수 있고 헛점이 많을 수 있는 점 양해 부탁드립니다.
오늘 제가 발표할 프로젝트는 '쉼표(,)' — 당신에게 필요한 휴식을 찾아드립니다 라는
개인 맞춤형 휴식 추천 플랫폼입니다.

프로젝트 이름을 쉼표로 정한 데에는 이유가 있습니다.
문장에서 쉼표는 "잠깐 멈추고, 다음으로 넘어가는 기호"입니다.
바쁜 일상 속에서 잠깐 멈추고 나에게 맞는 휴식을 취할 수 있도록 돕는 서비스입니다.

중간발표 이후 많은 기능이 추가됐는데요,
오늘은 최종 완성된 전체 서비스를 처음부터 끝까지 보여드리겠습니다.

오늘 발표 순서는 주제선정배경 → 벤치마킹 → 린캔버스 → 유스케이스 →
실제 구현 데모 → DB 설계 → 마무리 순으로 진행하겠습니다.

# [PART 2] 주제 선정 배경 (약 2분 30초)

(슬라이드: 주제선정배경 — 번아웃/시간부족/방법모름)

먼저 왜 이 주제를 선택했는지 말씀드리겠습니다.

현대인의 70% 이상이 번아웃을 경험한다는 통계가 있습니다.
다들 피로하다는 건 알고 있습니다. 그런데 막상 "쉬어야지"라고 생각하면서도 세
가지 문제에 막힙니다.

첫째, 시간이 없습니다. 직장인은 퇴근 후 몇 시간, 학생은 쉬는 시간 몇 분이 전부입니다.
둘째, 방법을 모릅니다. 쉬고 싶은데 뭘 해야 할지 모르겠다는 사람이 생각보다 훨씬 많습니다.
셋째, 기존 앱들은 추천이 없거나, 너무 비싸거나, 내 상태를 고려하지 않습니다.

(슬라이드: "쉬고 싶은데 어떻게 쉬어야 할지 모르겠어요")

이 슬라이드가 핵심입니다.
사용자에게 필요한 건 막연한 자유 시간이 아닙니다.
"지금 내 몸과 마음 상태에 맞는 휴식 처방" 이 필요합니다.

(슬라이드: 쉼표 개념 — 막연한 휴식을 구체적 '경험'으로)

그래서 쉼표는 세 가지를 지향합니다.
- 심리 기반 추천: 심리 테스트 + 심박수 분석으로 내 상태를 진단하고
- 자연 테마: 일상 속에서 접근 가능한 자연 기반 장소를 연결하며
- 즉시 실천 가능: 생각만 하는 게 아니라 바로 실행할 수 있는 가이드를 제공합니다.

# [PART 3] 벤치마킹 (약 2분)

(슬라이드: 벤치마킹 — 무디앱)

기획 단계에서 세 가지 서비스를 벤치마킹했습니다.

첫 번째는 무디앱입니다.
감성적인 UI와 기분 트래킹은 훌륭하지만, 단순 기록에 그칩니다.
"오늘 기분이 어때요?"는 물어보는데, "그럼 이렇게 해봐요"는 없습니다.
→ 쉼표는 여기서 '추천'과 '행동 가이드'를 추가합니다.

(슬라이드: 에어비앤비)

두 번째는 에어비앤비입니다.
카테고리별 큐레이션과 지역 기반 추천은 탁월합니다.
하지만 숙박/여행 중심이라 일상적인 휴식엔 맞지 않고, 금전적 부담도 큽니다.
→ 쉼표는 일상 속에서 무료 또는 저비용으로 접근 가능한 장소에 집중합니다.

(슬라이드: 핀터레스트)

세 번째는 핀터레스트입니다.
AI 기반 개인화 추천과 비주얼 영감 제공은 좋지만, 실행으로 이어지지 않습니다.
보다가 오히려 더 피로해지는 경험이 생깁니다.
→ 쉼표는 '보는 것'에서 '실행하는 것'으로 연결하는 단계를 핵심으로 설계했습니다.

이 세 서비스의 장점을 흡수하고 단점을 보완해서 쉼표의 핵심 기능을 도출했습니다.

# [PART 4] 린캔버스 (약 1분 30초)

(슬라이드: 린캔버스)

린캔버스로 비즈니스 모델을 정리했습니다.

문제: 번아웃 경험, 휴식 방법 부재, 기존 서비스의 한계
고객군: 직장인, 수험생, 육아 중인 부모, 크리에이터 — 스트레스 고위험군 전반입니다.
솔루션: 심박수와 심리 설문을 결합해 컨디션을 분석하고, 단계별 가이드를 제공합니다.
가치 제안: 단순 정보 제공이 아니라 "즉시 실행 가능한 휴식 처방" 입니다.
경쟁우위: 심박수 기반 실시간 분석과 실행 중심 콘텐츠 라이브러리, 커뮤니티 기능이 차별점입니다.
핵심 지표: 주간 활성 사용자 수, 추천 장소 클릭률, 서비스 이용 전후 감정 점수 변화입니다.

# [PART 5] 유스케이스 (약 1분)

(슬라이드: 유스케이스 다이어그램)

유스케이스 다이어그램으로 사용자와 관리자의 핵심 흐름을 정의했습니다.

사용자는 회원가입 및 로그인 또는 심리상태 진단을 통해 서비스에 진입하고,
→ 맞춤형 휴식 추천 → 장소 탐색 및 저장 → 휴식 기록으로 이어지며
→ 월간 통계와 개선 사이클로 마무리됩니다.
커뮤니티와 챌린지를 통해 같은 관심사의 사용자들과 연결되기도 합니다.

핵심 흐름은 진단 → 추천 → 활동 → 기록 → 개선 이 다섯 단계입니다.

# [PART 6] 실제 구현 데모 (약 8분)

## 1단계. 회원가입 & 로그인 (약 40초)

[행동] 주소창에 localhost:3000/signup 입력 또는 메인에서 회원가입 클릭

회원가입 화면입니다. 이메일, 비밀번호, 아이디를 입력하면 가입이 완료됩니다.
가입 시 쉼표번호가 자동 생성되고, 이메일 인증까지 완료하면 로그인이 가능합니다.

[행동] 이미 가입된 테스트 계정으로 바로 로그인 페이지로 이동 localhost:3000/login

실제로는 테스트 계정으로 바로 로그인하겠습니다.

[행동] 이메일: test@comma.com / 비밀번호: Test@1234 입력 → 로그인 버튼 클릭

로그인 성공 시 JWT 액세스 토큰이 발급되고 메인 대시보드로 이동합니다.
서버에서 발급된 토큰은 로컬스토리지에 저장되어 이후 모든 API 요청에 사용됩니다.

## 2단계. 메인 대시보드 (약 40초)

[행동] 로그인 후 메인 페이지 (localhost:3000/) 확인

메인 화면입니다. 상단에는 이번 달 휴식 횟수와 유형별 분포 통계가 한눈에 보이고,
그 아래로 심박수 체크와 휴식 유형 테스트 진입 카드가 있습니다.

[행동] 카테고리 탭 중 '자연의 연결', '정신적 휴식' 등 클릭해서 탭 전환 시연

가운데 휴식 카테고리 7종 탭을 클릭하면 해당 유형에 맞는 추천 콘텐츠와 장소로 필터링됩니다.

[행동] 추천 카드 중 하나 클릭해서 좋아요 버튼 토글 시연

좋아요 버튼을 누르면 즉시 저장됩니다.

[행동] 스크롤 내려서 맨 하단 맞춤 추천·추천 장소 영역 보여주기

맨 아래에는 진단 결과를 기반으로 생성된 맞춤 추천 카드와 추천 장소가 표시됩니다.

## 3단계. 심박수 측정 (약 50초)

[행동] 상단 심박수 체크 카드 클릭 → localhost:3000/heartrate 이동

심박수 측정 화면입니다.
Apple Watch 연동 모드와 시뮬레이션 모드 두 가지를 선택할 수 있습니다.

[행동] 측정 방식 중 "Apple Watch" 선택 → STEP 가이드 확인

Apple Watch 모드를 선택하면 iPhone 단축어 설정 가이드가 표시됩니다.
단축어를 한 번 만들어두면, 이후에는 단축어만 실행해도 BPM이 이 화면으로 전송됩니다.

[행동] "측정 시작" 버튼 클릭 → 측정 중 애니메이션 확인

측정이 완료되면 평균 BPM과 함께 세션 데이터가 DB에 저장됩니다.
이 데이터가 이후 진단 결과의 스트레스 지수 계산에 활용됩니다.

## 4단계. 휴식 유형 테스트 (약 1분)

[행동] 메인으로 돌아가서 "휴식 유형 테스트" 카드 클릭 → localhost:3000/rest-test 이동

휴식 유형 진단 테스트입니다. 총 12문항으로 구성되어 있습니다.

[행동] 진행 바 확인 후 1번 문항 선택지 아무거나 클릭

상단 진행 바로 현재 몇 번째 문항인지 확인할 수 있고,
선택지를 클릭하면 자동으로 다음 문항으로 넘어갑니다.

[행동] 빠르게 3~4문항 연속 클릭해서 흐름 보여주기

[행동] 12문항 모두 선택하거나, 이미 완료된 결과 페이지로 이동

12문항이 완료되면 7가지 휴식 유형 — 신체적 이완, 정신적 휴식, 감각적 휴식,
감정적 휴식, 사회적 휴식, 창조적 휴식, 자연의 연결 — 각각의 점수가 계산되어
나의 주요 휴식 유형이 도출됩니다.
심박수 데이터가 있으면 BPM 기반 스트레스 지수도 함께 반영됩니다.
유형별 점수와 스트레스 지수는 이후 장소 추천의 기준이 됩니다.

## 4-1단계. 스트레스 심화 진단 — PSS (약 30초)

[행동] 진단 결과 화면 하단 "더 정확한 스트레스 측정" 버튼 클릭 → localhost:3000/stress-test 이동

PSS-10은 Cohen(1983)이 개발한 국제 표준 스트레스 측정 도구입니다.
10문항, 5점 척도로 0~40점 범위로 스트레스를 수치화합니다.

[행동] 문항 2~3개만 빠르게 클릭해서 흐름 보여주기

결과가 나오면 낮음·보통·높음 세 단계로 분류됩니다.
로그인 사용자는 이 PSS 점수가 서버에 저장되어 장소 추천에 반영됩니다.

심박수 기반 스트레스 지수와 PSS 점수를 결합해 최종 스트레스 레벨을 계산하고,
스트레스가 높을수록 감정적 휴식·정신적 고요·자연 장소가 우선 추천되고
반대로 낮을수록 사교·창작 활동이 올라옵니다.

## 5단계. 휴식 기록 (약 1분)

[행동] 상단 네비게이션 → '기록' 클릭 → localhost:3000/rest-record 이동

휴식 기록 페이지입니다.

[행동] "기록 추가" 또는 기록 폼 영역 클릭

[행동] 휴식 유형 태그 중 '정신적' 또는 '자연의 연결' 클릭 → 선택 상태 확인

[행동] 시작 시간, 종료 시간 입력

[행동] 메모란에 "공원 산책 30분" 정도 간단히 입력

[행동] 휴식 전 감정 슬라이더와 후 감정 슬라이더를 각각 조절

휴식 전후 감정 점수를 1~10점으로 기록합니다.
이 데이터가 쌓이면 "내가 어떤 휴식을 했을 때 감정이 개선됐는지" 분석이 가능해집니다.

[행동] 저장 버튼 클릭 → 성공 토스트 메시지 확인

[행동] 저장 후 하단 월간 통계 영역 확인

월간 통계에서 이번 달 총 휴식 시간, 유형별 비율, 평균 감정 점수가 자동으로 집계됩니다.

## 6단계. 휴식 지도 (약 30초)

[행동] 상단 네비게이션 → '지도' 클릭 → localhost:3000/map 이동

휴식 지도 화면입니다.
공공데이터포털 기반 장소 데이터로 구성되어 있으며, 주변 휴식 장소 핀이 표시됩니다.

[행동] 좌측 카테고리 탭 클릭해서 필터 변경 → 장소 카드 목록 변화 확인

[행동] 지도 위 장소 핀 하나 클릭 → 카드 팝업 또는 사이드바 상세 표시 확인

카테고리 탭을 선택하면 해당 유형의 장소만 지도에 표시됩니다.
장소를 클릭하면 상세 정보 카드가 나타나고, 북마크 저장도 가능합니다.

## 7단계. 커뮤니티 (약 40초)

[행동] 상단 네비게이션 → '커뮤니티' 클릭 → localhost:3000/community 이동

커뮤니티 게시판입니다.
휴식 관련 경험을 자유롭게 공유할 수 있고, 익명으로 작성하는 것도 가능합니다.

[행동] 게시글 하나 클릭 → 상세 페이지 진입 → 댓글 영역, 좋아요 버튼 확인

[행동] 댓글 입력창에 짧게 입력 후 등록 시연 (또는 기존 게시글 좋아요 클릭)

게시글 작성 시 금칙어 자동 필터링이 동작합니다.
금칙어가 포함된 게시글은 작성자 본인에게만 보이는 숨김 상태로 처리됩니다.
신고 기능도 있어서 부적절한 게시글은 신고 후 관리자가 처리할 수 있습니다.

## 8단계. 챌린지 (약 30초)

[행동] 상단 네비게이션 → '챌린지' 클릭 → localhost:3000/challenge 이동

둘째, 추천 로그를 반드시 저장합니다.
사용자에게 어떤 장소를 추천했고, 클릭했는지, 저장했는지를 기록해야 통계와 개선 화면이 의미 있는 데이터를 보여줄 수 있습니다.

# Q&A 스크립트

## Q1. 심박수는 어떻게 측정해요? 카메라로 되나요?
A. 네, 스마트폰 카메라를 손가락으로 덮으면 빛의 반사량 변화로 심박수를 측정하는 방식입니다.
이걸 PPG(광혈류측정) 방식이라고 하는데, 손가락의 혈류량 변화를 카메라가 감지해서 BPM으로 변환합니다.
스마트워치도 같은 원리를 사용하고, 저는 이 기능을 웹에서 구현했습니다.
QR코드로 모바일 브라우저에서 접속하면 바로 측정할 수 있습니다.

## Q2. 지도는 어떤 API 사용했어요?
A. 공공데이터포털 API를 사용했습니다.
VWorld POI API로 공원, 도서관, 산책로 등 장소 데이터를 수집하고,
지도 렌더링은 OpenStreetMap과 Leaflet 라이브러리를 사용했습니다.
공공데이터를 활용하면 별도 비용 없이 신뢰도 높은 장소 정보를 사용할 수 있다는 장점이 있습니다.

## Q3. 카카오맵 말고 OpenStreetMap을 쓴 이유가 뭔가요?
A. 카카오맵 API는 사용량 제한과 유료 과금 구조가 있어서 학생 프로젝트 수준에서 지속적으로 쓰기가 부담스러웠습니다.
OpenStreetMap은 오픈소스 무료 지도고, Leaflet 라이브러리와 함께 쓰면 동일한 수준의 기능 구현이 가능합니다.
공공 데이터 기반 서비스라는 취지에도 더 잘 맞는다고 판단했습니다.

## Q4. 7가지 휴식 유형은 어떻게 정한 거예요?
A. 심리학에서 실제로 사용하는 휴식 유형 분류를 참고했습니다.
신체적 이완, 정신적 고요, 감각의 정화, 정서적 지지, 사회적 휴식, 자연의 연결, 창조적 몰입 — 이 7가지는 각각 다른 피로 원인에 대응하는 휴식 방식입니다.
설문 12문항의 답변 점수를 유형별로 평균 내어 가장 높은 유형을 메인으로 추천합니다.

## Q5. 혼자 다 만든 거예요?
A. 네, 기획부터 DB 설계, 백엔드, 프론트엔드까지 전부 혼자 진행했습니다.
백엔드는 Spring Boot와 MyBatis, 프론트엔드는 React와 Tailwind CSS를 사용했고,
DB는 MySQL, 인증은 JWT 기반으로 구현했습니다.
AI 이미지 분류는 YOLOv8, 지도는 Leaflet, 배포는 Google Cloud Run 예정입니다.

## Q6. 제일 어려웠던 부분이 뭐예요?
A. DB 설계가 가장 어려웠습니다.
처음에는 테이블을 단순하게 설계했는데, 기능이 늘어날수록 FK가 꼬이는 문제가 생겼습니다.
결국 원천 데이터와 산출물을 분리하는 방향으로 재설계했습니다.
심박수 측정값과 설문 응답은 원천 데이터로 보존하고, 진단 결과는 언제든 재계산이 가능하게 만들었습니다.
또 쉼표번호를 String PK로 설계한 부분도 처음엔 단순하게 생각했는데 FK가 27개 테이블에 걸쳐 있어서
나중에 고치기 어려운 상황이 됐습니다.

## Q7. Spring Security는 왜 안 썼어요?
A. Spring Security는 설정이 복잡하고, 이 프로젝트 규모에서는 오버엔지니어링이 될 수 있다고 판단했습니다.
대신 HandlerInterceptor 기반으로 JWT 인증을 직접 구현했고,
인증이 필요한 API에 대해 요청 헤더의 토큰을 검증하는 방식으로 처리했습니다.
구조가 단순해서 흐름을 이해하기 쉽고 커스터마이징도 자유롭습니다.

## Q8. YOLO가 뭐예요? 이미지 분석을 어디에 썼어요?
A. YOLO는 이미지를 실시간으로 분류하는 AI 모델입니다.
관리자가 장소를 승인할 때, 첫 번째 사진을 YOLO 서버에 보내면
"자연 속 휴식", "감각적 휴식" 같은 휴식 유형 태그가 자동으로 붙습니다.
관리자가 일일이 태그를 달지 않아도 돼서 운영 효율이 높아집니다.
Python FastAPI로 별도 서버를 구성했고, 꺼져 있어도 Spring Boot는 정상 동작합니다.

## Q9. 심박수 측정이 정확한가요?
A. 완전히 의료용 수준은 아니지만, 스트레스 지수를 판단하기에는 충분한 정확도를 갖고 있습니다.
평균 BPM과 HRV(심박변이도)를 함께 측정해서 단순 맥박이 아니라 자율신경계 상태까지 반영하도록 설계했습니다.
실제 스마트워치 앱들도 동일한 원리로 스트레스 지수를 산출합니다.

## Q10. 커뮤니티에서 금칙어 필터링은 어떻게 구현했어요?
A. 관리자가 금칙어를 등록하면 DB의 blocked_keywords 테이블에 저장됩니다.
게시글이나 댓글 작성 시 서버에서 해당 키워드 포함 여부를 확인해서, 포함된 경우 게시글을 즉시 차단하지 않고 작성자 본인에게만 보이는 숨김(hidden) 상태로 저장합니다.
거부 메시지를 보내면 사용자가 어떤 단어가 금칙어인지 역으로 파악할 수 있어서, 이 shadow-hide 방식을 선택했습니다.
관리자가 실시간으로 키워드를 추가하거나 삭제할 수 있고, 숨김 처리된 게시글도 관리자 화면에서 복원하거나 완전 삭제할 수 있습니다.

## Q11. 배포는 언제 할 예정이에요?
A. Google Cloud Run + Cloud SQL + Cloud Storage 기반으로 준비 중입니다.
Cloud Run은 컨테이너 기반 서버리스 배포 서비스로, 트래픽이 없을 때는 자동으로 내려가고
요청이 들어오면 즉시 올라오는 구조라 학생 프로젝트 규모에서 비용 효율이 좋습니다.
Docker로 백엔드를 컨테이너화해서 올리는 것이 마지막 남은 작업입니다.

## Q12. DB가 27개 테이블인데, 처음부터 다 설계한 건가요?
A. 처음부터 27개는 아니었고, 멘토링을 받으면서 계속 다듬었습니다.
"원천 데이터와 산출물을 분리하라", "추천 로그를 반드시 저장해야 통계가 살아난다"는 피드백을 받으면서 테이블이 늘어났습니다.
지금도 완전히 완성된 상태는 아니고, MVP 이후에 PK 구조 리팩토링이 예정되어 있습니다.
