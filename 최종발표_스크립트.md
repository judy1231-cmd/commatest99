**캐시(↔ 임시 저장소)**: Redis(↔ 빠른 메모리 DB) — 리프레시 토큰(↔ 로그인 연장 토큰) 저장

**배포**: Google Cloud Run(↔ 구글 서버 자동실행) + Cloud SQL + Cloud Storage(↔ 파일 저장소)

**지도**: Leaflet + OpenStreetMap(↔ 오픈소스 지도 데이터) — 카카오맵 대신 선택한 이유는 오픈소스 무료이고, 공공 데이터 취지에 부합하기 때문입니다.

`[슬라이드: 시스템 아키텍처(↔ 전체 구조) 다이어그램(↔ 흐름도)]`

전체 구조입니다.

사용자 브라우저 → React 화면 (포트 3000) → Spring Boot API 서버 (포트 8080) → MySQL 데이터베이스

심박수 측정은 모바일에서 별도 URL(↔ 주소)로 접속해 측정하고, REST API(↔ 서버 통신 방식)로 데이터를 전송합니다.

YOLO 서비스는 Python FastAPI(↔ 파이썬 서버 프레임워크) (포트 8090)로 분리 운영하며, 서버가 꺼져 있어도 메인 서비스는 정상 동작합니다.

특별히 언급할 설계 결정이 하나 있습니다.

**Spring Security(↔ 스프링 보안 프레임워크)를 사용하지 않았습니다.**

대신 HandlerInterceptor(↔ 요청 가로채기 클래스)를 직접 구현해서 JWT(↔ 로그인 토큰) 인증을 처리했습니다.

설정 복잡도를 낮추고, 이 프로젝트 규모에 맞는 가벼운 인증 구조를 선택했습니다.

## ① JWT 인증(↔ 로그인 토큰 검증) — HandlerInterceptor(↔ 요청 가로채기) 방식

`[슬라이드: JwtInterceptor 코드]`

```java
// JwtInterceptor.java
@Override
public boolean preHandle(HttpServletRequest request, ...) {
    String path = request.getServletPath();

    // 공개 경로 체크 (로그인, 회원가입 등)
    boolean isPublic = PUBLIC_PATHS 중 AntPathMatcher로 매칭;

    // 관리자 전용 경로 체크
    if (pathMatcher.match("/api/admin/**", path) && !"ADMIN".equals(role)) {
        sendError(response, 403, "관리자 권한이 필요합니다.");
        return false;
    }

    // JWT 검증 후 쉼표번호를 request attribute로 전달
    request.setAttribute("쉼표번호", 쉼표번호);
}
```

Spring Security(↔ 스프링 보안 프레임워크) 없이, 모든 인증과 인가(↔ 권한 확인)를 이 인터셉터(↔ 요청 중간 처리기) 하나가 처리합니다.

컨트롤러(↔ 요청 처리 클래스)에서는 `request.getAttribute("쉼표번호")`로 바로 사용자 정보를 꺼냅니다.

공개 경로라도 토큰(↔ 로그인 증명서)이 있으면 사용자 정보를 추출해서, 로그인 여부에 따라 다른 응답을 내려줄 수 있습니다.

## ② 진단 알고리즘(↔ 상태 계산 방식) — 심박수 + 설문 결합

`[슬라이드: DiagnosisService 코드]`

```java
// DiagnosisService.java
public DiagnosisResult calculate(String commaNo, Long sessionId) {

    // 1단계: 심박수 → 스트레스 지수 계산 (HRV 기반)
    double stressIndex = heartRateService.calculateStressIndex(sessionId);

    // 2단계: 설문 응답 → 유형별 평균 점수
    // 같은 유형 질문이 복수일 경우 평균값 사용
    Map<String, Double> typeScores = surveyService.calculateTypeScores(commaNo);

    // 3단계: 스트레스 지수가 높을수록 '신체 이완', '정신적 고요' 가중치 부여
    applyStressWeight(typeScores, stressIndex);

    // 4단계: DB 저장 — 원천(측정/응답)과 산출물(진단결과) 분리 보존
    return save(commaNo, sessionId, stressIndex, typeScores);
}
```

중요한 설계 원칙입니다.

심박수 측정값과 설문 응답은 원천 데이터(↔ 가공 전 데이터)로 따로 보존하고, 진단 결과는 언제든 재계산이 가능합니다.

HRV(↔ 심박 변이도, Heart Rate Variability)는 심박수 간격의 변화량으로, 숫자가 낮을수록 스트레스 상태를 의미합니다.

## ③ 추천 알고리즘(↔ 추천 계산 방식) — 3단계 개인화

`[슬라이드: RecommendationMapper.xml 코드]`

추천은 세 단계가 순서대로 적용됩니다.

**1단계 — 개인 효과 이력**

```sql
-- 내가 최근 30일간 어떤 휴식 유형에서 감정이 가장 많이 개선됐는지
SELECT rt.type_name,
       AVG(rl.emotion_after - rl.emotion_before) AS avg_improvement
FROM rest_logs rl
JOIN rest_types rt ON rl.rest_type_id = rt.id
WHERE rl.`쉼표번호` = #{commaNo}
  AND rl.start_time >= DATE_SUB(NOW(), INTERVAL 30 DAY)
GROUP BY rt.type_name
```

**2단계 — 협업 필터링(↔ 유사 사용자 기반 추천)**

나와 같은 유형을 2개 이상 사용한 '유사 사용자'를 찾고,
그 사람들의 평균 감정 향상도를 참고합니다.
데이터가 쌓일수록 추천 정확도가 높아지는 구조입니다.

**3단계 — 최적 패턴 학습**

```sql
-- 요일 × 시간대별로 어떤 유형이 효과적이었는지
-- "이 사람은 월요일 저녁에 자연 휴식이 가장 효과적이다"
WHERE DAYOFWEEK(start_time) = #{dayOfWeek}
  AND HOUR(start_time) BETWEEN 18 AND 21  -- 저녁 시간대
GROUP BY rt.type_name
HAVING COUNT(*) >= 2  -- 샘플 2개 이상만 신뢰
```

단순 진단 결과 기반 추천이 아닌, **내 기록이 쌓일수록 더 정확해지는 알고리즘**입니다.

## ④ YOLO(↔ 인공지능 이미지 분류) — 장소 자동 태깅

`[슬라이드: YOLO 연동 흐름도]`

관리자가 장소를 승인할 때, 첫 번째 사진을 Python YOLO 서버로 자동 전송합니다.

```java
// AdminService.java — 장소 승인 시 자동 호출
public void updatePlaceStatus(String adminNo, Long placeId, String status) {
    adminMapper.updatePlaceStatus(placeId, status);
    if ("approved".equals(status)) {
        // 사진 → YOLO 서버 전송 → 휴식 유형 태그 자동 등록
        yoloService.autoTagByYolo(placeId);
    }
}
```

YOLO 서버가 꺼져 있어도 승인 자체는 정상 동작합니다.

이걸 그레이스풀 디그레데이션(↔ Graceful Degradation, 우아한 성능 저하) 설계라고 하는데, 외부 서비스에 장애가 생겨도 메인 기능은 살아있도록 방어하는 방식입니다.

# [PART 6] DB 설계 ⏱ 1분 30초

`[슬라이드: ERD(↔ 데이터 구조도) — 27개 테이블, 7개 도메인]`

DB(↔ 데이터베이스, 데이터 저장소)는 총 27개 테이블, 7개 도메인(↔ 기능 영역)으로 설계했습니다.

**핵심 데이터 흐름입니다:**

```
users (사용자)
  → measurement_sessions (심박 측정 세션)
    → heart_rate_measurements (원천: BPM, HRV 측정값)
  → survey_responses (원천: 설문 응답)
    → diagnosis_results (산출물: 진단 결과)
      → rest_type_scores (유형별 점수)
        → recommendations (장소 추천 로그)
  → rest_logs (휴식 기록)
    → monthly_stats (월간 통계 집계)
```

중요한 설계 원칙 두 가지를 말씀드립니다.

**첫째, 원천 데이터와 산출물 분리.**

심박수 측정값과 설문 응답은 원천 그대로 보존합니다. 진단 결과는 언제든 재계산이 가능합니다.

**둘째, 추천 로그(↔ 기록) 반드시 저장.**

recommendations(↔ 추천 기록) 테이블에 어떤 장소를 추천했고, 클릭했는지, 저장했는지를 기록합니다. 이 로그가 없으면 추천 알고리즘 성능을 측정할 수 없습니다.

# [PART 7] 추천 알고리즘 심화 ⏱ 2분

`[슬라이드: 추천 알고리즘 3단계 플로우차트(↔ 흐름도)]`

쉼표의 추천 알고리즘을 좀 더 깊이 설명드리겠습니다.

총 3단계가 순서대로 적용됩니다.

**1단계 — 개인 효과 이력 (30일)**

내가 최근 30일 동안 어떤 휴식 유형을 했을 때 감정 점수가 가장 많이 올랐는지 계산합니다. 데이터가 있을 경우 가장 강력한 기준입니다.

**2단계 — 협업 필터링(↔ 유사 사용자 참고) (90일)**

나와 비슷한 사람들 — 같은 휴식 유형을 2개 이상 공유하는 사용자들 — 의 평균 감정 향상도를 참고합니다.

내 기록이 부족할 때 보완해주는 역할입니다. 샘플(↔ 데이터 수) 3개 미만인 유형은 신뢰도가 낮아 제외합니다.

**3단계 — 요일×시간대 패턴**

"이 사람은 월요일 저녁에 자연 휴식이 가장 효과적이었다"는 패턴을 학습합니다.

샘플 2개 이상 쌓인 경우에만 적용해 초기 데이터 부족 문제를 방어합니다.

`[슬라이드: 알고리즘 성장 그래프 — 데이터 축적에 따른 정확도 향상]`

이 구조의 핵심은 **"쓸수록 더 똑똑해진다"** 입니다.

처음 가입한 사용자는 진단 결과 기반 추천을 받고, 기록이 쌓이면 개인 이력 기반, 나아가 요일·시간대 패턴까지 반영됩니다.

# Q&A 대본

> 모르는 질문이 나오면:
> **"좋은 질문입니다. 현재는 ~~까지 구현했고, 해당 부분은 추후 개선 예정입니다."**

---

**Q1. Spring Security(↔ 스프링 보안 프레임워크)를 왜 안 쓴 거예요?**

이 프로젝트 규모에서는 오버엔지니어링(↔ 필요 이상의 복잡한 구현)이 될 수 있다고 판단했습니다.

HandlerInterceptor(↔ 요청 가로채기 클래스) 기반으로 JWT(↔ 로그인 토큰) 인증을 직접 구현했고, 공개 경로와 관리자 전용 경로를 명확히 분리했습니다.

오히려 구조가 더 단순해져서 디버깅(↔ 오류 찾기)하기 쉬웠습니다.

---

**Q2. 추천 알고리즘(↔ 추천 계산 방식)은 처음부터 세 단계였나요?**

아닙니다. 처음엔 진단 결과만 보고 추천하는 단순한 구조였습니다.

기록이 쌓여야 의미 있는 추천이 가능하다는 걸 깨닫고, 개인 이력 → 협업 필터링(↔ 유사 사용자 참고) → 패턴 학습 순으로 단계를 늘렸습니다.

---

**Q3. 심박수 측정이 얼마나 정확해요?**

의료용 수준은 아니지만, 스트레스 지수 판단에는 충분합니다.

BPM(↔ 분당 심박수)과 HRV(↔ 심박 변이도)를 함께 측정해서 자율신경계 상태를 반영합니다.

Apple Watch, Samsung Health도 동일한 PPG(↔ 광혈류측정) 방식을 사용합니다.

---

**Q4. 지도 데이터는 어디서 가져왔어요?**

국내는 공공데이터포털에서 공원, 도서관, 산책로 데이터를 Python(↔ 파이썬)으로 크롤링(↔ 자동 수집)해서 DB에 저장했습니다.

해외 장소는 직접 큐레이션(↔ 선별 입력)해서 입력했습니다.

지도 렌더링(↔ 화면 표시)은 Leaflet + OpenStreetMap(↔ 오픈소스 지도)을 사용했고, 카카오맵 대신 선택한 이유는 오픈소스 무료이고 사용량 제한이 없기 때문입니다.

---

**Q5. YOLO(↔ 이미지 분류 AI)는 왜 넣었어요?**

장소 사진을 관리자가 일일이 유형 태깅(↔ 분류 태그 달기)하는 건 비효율적입니다.

장소를 승인할 때 자동으로 사진을 분석해서 '자연', '카페', '운동시설' 같은 태그를 붙여줍니다.

서버가 꺼져 있어도 승인 자체는 정상 동작하도록 그레이스풀 디그레데이션(↔ Graceful Degradation, 장애 상황 대비 설계)으로 만들었습니다.

---

**Q6. DB가 27개 테이블인데 처음부터 설계한 건가요?**

아닙니다. 멘토링을 통해 계속 다듬었습니다.

"원천 데이터와 산출물을 분리하라", "추천 로그를 저장해야 통계가 살아난다"는 피드백이 설계를 크게 바꿨습니다.

---

**Q7. 혼자 다 만든 건가요?**

기획부터 DB 설계, 백엔드(↔ 서버), 프론트엔드(↔ 화면), AI 연동까지 전부 혼자 진행했습니다.

Claude Code(↔ AI 코딩 도구)를 활용해서 개발 속도를 높였습니다.

---

**Q8. 가장 어려웠던 부분이 뭔가요?**

DB 설계가 가장 어려웠습니다.

기능이 늘어날수록 FK(↔ 외래키, 테이블 간 연결) 관계가 복잡해졌고, 결국 원천 데이터와 산출물을 분리하는 방향으로 전면 재설계했습니다.

처음부터 잘 설계하는 것과 나중에 고치는 것의 비용 차이를 직접 경험했습니다.

---

**Q9. 배포는 됐나요?**

Google Cloud Run(↔ 구글 클라우드 서버 자동 실행) + Cloud SQL(↔ 클라우드 DB) + Cloud Storage(↔ 클라우드 파일 저장소) 구조로 배포 준비가 완료된 상태입니다.

환경변수(↔ 서버 설정값) 설정 후 `gcloud run deploy` 명령 한 번으로 배포됩니다.
